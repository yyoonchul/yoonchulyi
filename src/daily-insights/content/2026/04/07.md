## 📝 Detailed Notes

### 1. LLM Wiki

1. The note starts by rejecting the usual "upload files and retrieve chunks later" pattern as too stateless.
   - Karpathy says standard RAG workflows force the model to rediscover useful fragments from scratch on every question.
   - That means synthesis is repeatedly recomputed instead of stored, refined, and extended over time.
   - The proposed alternative is a persistent markdown wiki that sits between the user and the raw document collection.
2. The central artifact is a wiki the LLM actively maintains rather than a notebook the human updates manually.
   - When a new source arrives, the model is supposed to summarize it, update relevant concept pages, revise entity pages, and note contradictions with earlier material.
   - The emphasis is on accumulation: cross-references, synthesized claims, and open questions should already exist before the next query happens.
   - In the author's framing, the human curates sources and asks questions while the LLM handles the filing and bookkeeping.
3. The architecture is intentionally simple and split into three layers with clear responsibilities.
   - Raw sources are immutable and remain the source of truth that the LLM reads but does not edit.
   - The wiki is a directory of generated markdown pages covering summaries, entities, concepts, and higher-level syntheses.
   - A schema document such as `AGENTS.md` or `CLAUDE.md` tells the model how to structure the wiki and how to behave during ingest, query, and maintenance workflows.
4. The operational model treats ingestion, querying, and linting as first-class recurring workflows.
   - Ingest can touch many pages from one source because the goal is to merge new material into an existing knowledge graph rather than file one isolated summary.
   - Query answers can themselves be written back into the wiki so valuable analysis does not disappear into chat history.
   - Lint passes are meant to catch stale claims, orphan pages, missing links, contradictions, and places where new research is needed.
5. The pitch is ultimately about lowering the maintenance cost that usually kills personal or team knowledge bases.
   - The note argues humans abandon wikis because cross-linking and consistency updates create too much repetitive overhead.
   - LLMs are presented as a good fit for exactly that tedious work: touching many markdown files, updating references, and keeping summaries synchronized.
   - Obsidian, markdown, indexing files, and optional search tools are framed as practical implementation choices, but the main idea is the compounding wiki pattern itself.

### 2. I Still Prefer MCP Over Skills

1. Mohl's opening claim is that the current "Skills over MCP" narrative confuses two very different jobs.
   - He is comfortable with Skills when they teach an LLM how to format output, use existing tools, or follow organization-specific conventions.
   - His objection is to using Skills plus ad hoc CLIs as the default way to grant real service access.
   - In his framing, that substitutes manuals and wrappers for a proper integration interface.
2. He argues MCP gets the connector layer right because it abstracts execution behind a service-owned tool interface.
   - A remote MCP can expose capabilities without requiring local installs on every client that wants to use the service.
   - Updates propagate at the server layer, authentication can be handled more cleanly, and the same connector can work across Mac, web, or mobile contexts.
   - He also highlights sandboxing and tool discovery as advantages over giving the model arbitrary shell access to a purpose-built CLI.
3. The critique of Skills becomes strongest when a Skill assumes the user can install and maintain a custom command-line tool.
   - Mohl points out that many common LLM environments cannot run CLIs at all, which makes those Skills dead on arrival outside full coding agents.
   - Even when a CLI can run, he sees packaging, installation, version skew, and token handling as unnecessary friction compared with a remote MCP.
   - He also criticizes context bloat: loading a long `SKILL.md` to invoke one action is a worse interface than exposing a focused tool signature.
4. The essay still defends Skills, but only for narrower knowledge and workflow roles.
   - Teaching an agent how to use `curl`, `git`, `gh`, or internal business jargon is presented as a strong fit for Skills.
   - He also likes repo-local Skills that explain secrets workflows, domain-specific conventions, and the quirks of tools already available in the environment.
   - In that model, Skills are closer to operational guidance documents than execution substrates.
5. His final position is a layered hybrid rather than a winner-take-all replacement.
   - MCP should own the connector problem for websites, applications, and services that need structured, authenticated access.
   - Skills should sit on top as a cheat sheet that records gotchas, common patterns, and tool-usage advice discovered during real work.
   - That is why he ends up advocating for both: connectors for action, manuals for context.

   - 그 결과, 종합적 이해는 저장되고 발전되는 대신 매번 다시 계산됩니다.
   - 그 대안으로 제시되는 것은 사용자와 원천 문서 사이에 놓이는 지속형 마크다운 위키입니다.
2. 핵심 산출물은 사람이 손수 관리하는 노트북이 아니라 LLM이 적극적으로 유지하는 위키입니다.
   - 새로운 소스가 들어오면 모델은 그것을 요약하고, 관련 개념 페이지를 갱신하고, 엔터티 페이지를 수정하고, 이전 자료와 충돌하는 부분을 표시해야 합니다.
   - 초점은 축적에 있습니다. 다음 질문이 들어오기 전에 교차 링크, 종합된 주장, 열린 질문이 이미 위키 안에 존재해야 한다는 뜻입니다.
   - 저자의 구도에서는 사람은 소스를 큐레이션하고 질문을 던지며, LLM은 정리와 장부 맞추기를 담당합니다.
3. 아키텍처는 의도적으로 단순하며, 역할이 분명한 3개 계층으로 나뉩니다.
   - Raw sources는 변경하지 않는 진실의 근원이며, LLM은 읽기만 하고 수정하지 않습니다.
   - Wiki는 요약, 엔터티, 개념, 상위 수준의 종합 페이지를 담는 생성형 마크다운 디렉터리입니다.
   - `AGENTS.md`나 `CLAUDE.md` 같은 스키마 문서는 위키 구조와 수집·질의·유지보수 시 모델이 따라야 할 행동 규칙을 정의합니다.

   - Query 결과도 다시 위키에 기록될 수 있어, 가치 있는 분석이 채팅 기록 속으로 사라지지 않게 합니다.

   - 반대로 LLM은 여러 마크다운 파일을 한 번에 만지고, 참조를 갱신하고, 요약을 동기화하는 지루한 일을 맡기에 적합한 존재로 제시됩니다.
   - Obsidian, 마크다운, 인덱스 파일, 선택적 검색 도구는 구현 수단일 뿐이고, 핵심 아이디어는 축적형 위키 패턴 자체입니다.

### 2. I Still Prefer MCP Over Skills

1. Mohl의 도입부 주장은 현재의 "Skills가 MCP를 대체한다"는 내러티브가 서로 다른 두 일을 혼동한다는 것입니다.
   - 그는 Skills가 출력 형식을 가르치거나, 기존 도구 사용법을 설명하거나, 조직 고유의 규칙을 전달하는 용도로는 충분히 유용하다고 봅니다.
   - 문제는 실제 서비스 접근 권한을 주는 기본 방식으로 Skills와 임시방편 CLI를 쓰려는 흐름입니다.
   - 그의 관점에서 그것은 제대로 된 통합 인터페이스 대신 설명서와 래퍼를 쌓는 일입니다.
2. 그는 MCP가 connector 계층을 더 잘 설계한다고 주장합니다. 실행을 서비스 소유의 도구 인터페이스 뒤로 숨기기 때문입니다.
   - 원격 MCP는 기능을 노출하면서도, 서비스를 쓰려는 모든 클라이언트에 로컬 설치를 강요하지 않습니다.
   - 업데이트는 서버 계층에서 전파되고, 인증은 더 깔끔하게 처리될 수 있으며, 같은 connector가 Mac, 웹, 모바일 맥락을 가로질러 동작할 수 있습니다.
   - 그는 이런 방식이 전용 CLI를 셸에서 마구 실행하게 하는 접근보다 샌드박싱과 도구 탐색 측면에서도 낫다고 봅니다.
3. Skills 비판은 Skill이 사용자가 커스텀 CLI를 설치하고 유지할 수 있다고 가정할 때 가장 강해집니다.
   - Mohl은 많은 LLM 환경이 아예 CLI를 실행할 수 없기 때문에, 그런 Skills는 코딩 에이전트 밖에서는 시작부터 무용하다고 지적합니다.
   - CLI를 실행할 수 있는 경우에도 그는 패키징, 설치, 버전 불일치, 토큰 관리가 원격 MCP보다 불필요한 마찰이라고 봅니다.
   - 또한 한 번의 동작을 위해 긴 `SKILL.md`를 통째로 컨텍스트에 싣는 것은, 집중된 도구 시그니처를 노출하는 것보다 나쁜 인터페이스라고 비판합니다.
4. 그렇다고 이 글이 Skills 자체를 부정하는 것은 아니며, 오히려 더 좁고 적절한 역할을 옹호합니다.
   - `curl`, `git`, `gh` 사용법이나 조직 내부 용어를 가르치는 일은 Skills에 잘 맞는 사례로 제시됩니다.
   - 그는 비밀 관리 워크플로, 도메인 규칙, 이미 설치된 도구의 특이점을 설명하는 저장소 로컬 Skills도 높게 평가합니다.
   - 이 모델에서 Skills는 실행 기반이 아니라 운영 가이드를 담은 문서에 더 가깝습니다.
5. 최종 입장은 승자독식 대체가 아니라 계층형 결합입니다.
   - MCP는 웹사이트, 애플리케이션, 서비스에 구조화되고 인증된 접근을 제공하는 connector 문제를 담당해야 합니다.
   - Skills는 그 위에 놓여, 실제 작업 중 발견한 함정, 자주 쓰는 패턴, 도구 사용 팁을 정리하는 치트시트 역할을 해야 합니다.
   - 그래서 그의 결론은 둘 다 필요하다는 것입니다. 행동에는 connector가, 맥락에는 manual이 필요하다는 뜻입니다.
