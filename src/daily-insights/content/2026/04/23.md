## 📝 Detailed Notes

### 1. Agent Vault: The Open Source Credential Proxy and Vault for Agents

1. The article starts from a concrete failure mode in current agent architecture.
   - Tony Dang argues that modern secret-delivery patterns were designed for deterministic software, not for agents that can be redirected or manipulated mid-run.
   - In his framing, the core issue is not merely secret storage but secret exposure during execution, when an agent can be prompt-injected into leaking credentials.
   - That shifts the problem from "how do we store secrets safely?" to "how do we let an agent act with privileges without ever possessing the secret itself?"
2. Existing mitigations are presented as useful but fundamentally incomplete.
   - The piece says many teams already rely on short-lived credentials, automated rotation, and dynamic secret issuance to narrow the blast radius of compromise.
   - It also cites more custom proxy patterns from teams such as Anthropic and Browser Use, plus provider-specific brokering products from companies like Vercel and Cloudflare.
   - Dang's point is that all of those examples indicate an industry-wide convergence: classic vault-to-workload delivery is a weak fit for agents.
3. Agent Vault's core design is a trust boundary between the agent and the credential.
   - The project is described as an open-source research-preview HTTP forward proxy plus vault, built on the principle that the agent should never directly see the underlying secret.
   - Instead, the secret lives in the vault and is attached only when outbound traffic passes through the proxy layer on its way to the target service.
   - That means the agent keeps its workflow while losing direct custody of the sensitive material it would otherwise be tempted or tricked into exfiltrating.
4. The implementation is intentionally interface-agnostic.
   - The article emphasizes that APIs, CLIs, SDKs, and MCP tools all eventually collapse into outbound HTTPS requests, so the credential-brokering layer should sit below the application interface.
   - Agent Vault therefore uses `HTTPS_PROXY`, terminates TLS locally, matches outbound requests to configured services, injects the right credential, and then re-establishes the upstream TLS connection.
   - The design also introduces vaults, credentials, services, and agents as separate constructs so teams can scope access more tightly and keep policy consistent across tools.
5. The launch is framed as a directional prototype, not a finished security product.
   - Dang says the project ships today as a portable binary and Docker-deployable service that can run locally next to coding agents or near sandboxed agent infrastructure.
   - He also acknowledges rough edges around ergonomics, security hardening, and operational setup, and explicitly calls the current release a research preview with significant changes still expected.
   - The strategic bet is nevertheless clear: credential brokering through a dedicated proxy is presented as the next default mental model for agent-era secrets management.

### 6. A new way to think about composing skills to increase leverage: Skill Graphs 2.0

1. Shiv starts by separating the appealing idea of skill composition from the brittle way many people currently implement it.
   - He describes the popular "skill graph" pattern as linking dependent markdown skills together, similar to how knowledge notes are connected in Obsidian.
   - That looks elegant at small scale because it mirrors how larger workflows really do depend on smaller reusable procedures.
   - The problem, he says, is that intuition breaks once the graph becomes deep and densely connected.
2. The failure mode is predictability, not just raw capability.
   - Shiv argues that agents become less reliable at invoking skills beyond a certain dependency depth, especially when composition chains grow long or circular.
   - In a simple Skill A to Skill B case, he thinks execution can remain dependable, but a wiki-like graph introduces too much non-determinism for the human operator.
   - That means users hand off more judgment than they realize while losing the ability to forecast what the agent will actually do.
3. His replacement is a hierarchy of atoms, molecules, and compounds.
   - Atoms are narrow single-purpose primitives such as scraping profiles, researching a topic, or reviewing a PR.
   - Molecules are scoped workflows or orchestrators that combine a handful of atoms with explicit instructions about when and how to use them.
   - Compounds sit above that as higher-level playbooks that coordinate multiple molecules to execute a broader business or product objective.
4. The model is also a theory of where humans should stay in the loop.
   - Shiv says atoms should be near-deterministic and usually should not call other skills at all, while molecules still try to push composition into the skill rather than leaving decisions fully to runtime.
   - Compounds, by contrast, are where real autonomy appears and where he still expects human guidance to matter because the agent is making more consequential judgments.
   - So the hierarchy is not only about software structure; it is also about assigning the right level of ambiguity to each layer.
5. The payoff he cares about is leverage against limited human attention.
   - Shiv describes "brain RAM" as the scarce resource, arguing that a human can only context-switch across a limited number of agent threads at once.
   - If compounds reliably orchestrate molecules and atoms, then the same supervisory bandwidth can unlock far more total work than directly driving atomic tasks one by one.
   - He closes with an open caveat, though: reliability at every layer is still hard to test, and he suspects compounds spanning more than roughly eight to ten molecules may hit their own ceiling.

2. 기존 완화책은 도움이 되지만 근본 해법은 아니라는 입장입니다.
   - 글은 많은 팀이 이미 short-lived credential, 자동 rotation, dynamic secret issuance 같은 방법으로 손상 범위를 줄이려 한다고 설명합니다.
   - 동시에 Anthropic, Browser Use, Vercel, Cloudflare 같은 사례를 들어 proxy 또는 credential brokering 패턴이 이미 여러 곳에서 등장하고 있다고 짚습니다.
   - Dang의 요지는 이 모든 사례가 하나의 방향을 가리킨다는 것입니다. 전통적인 vault-to-workload 전달은 에이전트에는 약한 모델이라는 신호입니다.
3. Agent Vault의 핵심 설계는 에이전트와 credential 사이에 trust boundary를 두는 것입니다.
   - 이 프로젝트는 에이전트가 실제 시크릿을 보지 않도록 설계된 오픈소스 research preview HTTP forward proxy 겸 vault로 소개됩니다.
   - 시크릿은 vault 안에 남아 있고, outbound 요청이 target service로 나갈 때 proxy 레이어에서만 붙습니다.
   - 즉 에이전트는 기존 워크플로를 유지하되, 유출되기 쉬운 민감 정보를 직접 손에 쥐지 않게 됩니다.
4. 구현은 인터페이스 종류와 무관하게 동작하도록 설계됐습니다.
   - 글은 API, CLI, SDK, MCP tool이 결국 outbound HTTPS 요청으로 수렴하므로, credential brokering도 애플리케이션 인터페이스 아래 계층에 있어야 한다고 강조합니다.
   - 그래서 Agent Vault는 `HTTPS_PROXY`를 사용하고, 로컬에서 TLS를 종료한 뒤, 서비스별 설정에 맞춰 적절한 credential을 붙이고, 다시 upstream TLS 연결을 맺습니다.
   - 또한 vault, credential, service, agent를 분리된 구성요소로 정의해 더 촘촘한 접근 범위 제어와 일관된 정책 적용을 가능하게 합니다.
5. 이 출시는 완성형 보안 제품이 아니라 방향성을 보여 주는 프로토타입으로 제시됩니다.
   - Dang은 이 프로젝트가 today 기준으로 portable binary와 Docker 배포 형태로 제공되며, 로컬 coding agent 옆이나 sandboxed agent 인프라 가까이에 둘 수 있다고 설명합니다.
   - 동시에 ergonomics, security hardening, operational setup 측면에서 거칠고 앞으로 크게 바뀔 여지가 많다고 공개적으로 인정합니다.
   - 그럼에도 전략적 베팅은 분명합니다. 전용 proxy를 통한 credential brokering이 agent 시대의 기본적인 시크릿 관리 모델이 되어야 한다는 주장입니다.

### 5. Stitch app’s DESIGN.md format is now open-source for designers

1. Google의 발표는 표면적으로는 짧지만 겨냥하는 레이어가 중요합니다.
   - 이 글은 Stitch의 `DESIGN.md`가 프로젝트 간 디자인 규칙을 export/import할 수 있게 해 주므로, 디자인 시스템을 매번 처음부터 다시 만들 필요가 없다고 설명합니다.
   - 즉 이것은 단순 style token 묶음이 아니라, Stitch가 인터페이스를 생성할 때 해석할 수 있는 portable한 design intent 표현입니다.
   - Google은 결국 디자인 rationale 자체를 재사용 가능한 machine-readable asset으로 만들려는 셈입니다.
2. 글은 이 릴리스를 AI agent usability와 직접 연결합니다.
   - Cassia Xu는 `DESIGN.md` 덕분에 AI agent가 화면 결과만 보고 색상과 컴포넌트의 의미를 추측하지 않아도 된다고 말합니다.
   - 대신 구조화된 규칙을 읽고, 그 규칙을 기반으로 결정하며, WCAG 접근성 기준까지 검증할 수 있다는 설명입니다.
   - 실무적으로는 기존 UI 시스템을 확장하거나 수정할 때 agent가 덜 애매한 상태에서 일할 수 있다는 뜻입니다.
3. 포맷을 오픈소스화했다는 점이 제품 기능에서 생태계 primitive로 범위를 넓힙니다.
   - Google은 `DESIGN.md`가 한 도구나 플랫폼 안에만 머물지 않고 경계를 넘어 쓰일 수 있도록 draft specification을 공개한다고 말합니다.
   - 이는 특히 AI 생성이 끼어드는 순간 더 쉽게 깨지는 cross-tool interoperability 문제를 겨냥한 선택으로 보입니다.
   - 다시 말해 지금 단계에서는 proprietary lock-in보다 shared semantics가 더 중요하다는 베팅입니다.
4. 발표에는 이론만이 아니라 바로 써 볼 수 있는 도입 경로도 포함돼 있습니다.
   - 글은 Stitch에서 파일을 생성하는 경로와 공개 GitHub 저장소 기여 링크를 함께 제공합니다.
   - 또 Google Labs의 David East가 설명하는 영상을 연결해, 이 개념이 개발자와 디자이너 모두에게 어느 정도 온보딩이 필요하다는 점도 드러냅니다.
   - 그래서 이것은 단순한 연구 예고가 아니라, 팀이 당장 열어 보고 실험할 수 있는 형태로 나온 릴리스에 가깝습니다.
5. 더 큰 의미는 디자인 시스템을 소프트웨어 에이전트가 읽을 수 있게 만든다는 데 있습니다.
   - `DESIGN.md`가 의도대로 작동하면 agent는 브랜드 규칙, 접근성 제약, 컴포넌트 의미를 수정 작업 전반에서 더 안정적으로 보존할 수 있습니다.
   - 이는 많은 현재 AI 디자인 워크플로가 여전히 화면을 느슨한 프롬프트나 스크린샷 수준으로 취급해 UI의 "왜"를 잃어버리는 문제와 대비됩니다.
   - Google의 메시지는 rationale이 명시적이고 portable해지는 순간, agentic design tool이 단순 모사에서 system-aware editing으로 넘어갈 수 있다는 것입니다.

### 6. A new way to think about composing skills to increase leverage: Skill Graphs 2.0

1. Shiv는 먼저 skill composition 자체의 가치와, 사람들이 지금 흔히 구현하는 방식의 취약함을 분리합니다.
   - 그는 skill graph 패턴을 의존 관계가 있는 markdown skill들을 서로 연결하는 방식으로 설명하며, Obsidian의 노트 링크에 비유합니다.
   - 이런 구조는 더 큰 워크플로가 더 작은 재사용 가능한 절차에 의존한다는 현실과 잘 맞기 때문에, 작은 규모에서는 매우 자연스럽게 보입니다.
   - 하지만 그의 말에 따르면 그래프가 깊고 촘촘해지는 순간 그 직관은 무너집니다.
2. 실패 지점은 성능 그 자체보다 예측 가능성입니다.
   - Shiv는 dependency depth가 깊어질수록 에이전트가 그 아래쪽 skill을 안정적으로 호출하지 못한다고 주장합니다.
   - 단순한 Skill A to Skill B는 괜찮을 수 있지만, 위키 같은 밀집 그래프는 인간 운영자가 실제 실행을 예측하기 어렵게 만듭니다.
   - 결국 사용자는 자신이 생각하는 것보다 더 많은 판단을 에이전트에 넘기면서도, 결과를 통제할 근거는 잃게 됩니다.
3. 그가 제안하는 대안은 atoms, molecules, compounds라는 계층 구조입니다.
   - Atoms는 프로필 스크래핑, 토픽 리서치, PR 리뷰 같은 좁고 단일 목적의 primitive입니다.
   - Molecules는 몇 개의 atom을 명시적으로 연결해 더 큰 scoped task를 해결하는 workflow 또는 orchestrator입니다.
   - Compounds는 그 위에서 여러 molecule을 조합해 더 넓은 비즈니스나 제품 목표를 수행하는 상위 playbook입니다.
4. 이 모델은 인간이 어디에 남아 있어야 하는지에 대한 이론이기도 합니다.
   - Shiv는 atom은 거의 deterministic해야 하며 보통 다른 skill을 다시 호출하지 말아야 한다고 봅니다. molecule 역시 가능한 한 composition을 skill 안에 밀어 넣어 runtime 판단을 줄여야 한다고 말합니다.
   - 반면 compound는 실제 자율성이 나타나는 층으로, 더 큰 판단이 필요한 만큼 여전히 인간의 운전이 중요하다고 설명합니다.
   - 그래서 이 계층은 단순한 코드 구조가 아니라, 각 레벨에 어느 정도의 ambiguity를 허용할지를 정하는 운영 모델이기도 합니다.
5. 그가 궁극적으로 노리는 보상은 제한된 인간 주의력을 넘어서는 레버리지입니다.
   - Shiv는 병목 자원을 "brain RAM"으로 설명하며, 사람은 동시에 관리할 수 있는 agent thread 수가 제한돼 있다고 봅니다.
   - Compound가 molecule과 atom을 안정적으로 orchestration해 준다면, 같은 감독 대역폭으로도 atomic task를 하나씩 직접 몰 때보다 훨씬 많은 총량의 일을 처리할 수 있습니다.
   - 다만 그는 열린 caveat도 남깁니다. 각 레이어의 신뢰성을 검증하는 일은 여전히 어렵고, 8개에서 10개 정도를 넘는 molecule을 다루는 compound는 또 다른 한계에 부딪힐 수 있다고 추정합니다.
