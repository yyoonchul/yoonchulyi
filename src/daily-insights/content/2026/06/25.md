## 📝 Detailed Notes

### 1. Mixing visual and textual code

1. The paper argues that purely textual programming is insufficient for some domains.
   - The authors focus on geometric and interactive ideas that are awkward to encode as linear text alone.
   - Visual programming languages help with those domains, but many require the whole program to become visual.
   - The proposed alternative is a hybrid language where text and visual constructs coexist on equal footing.
2. Hybrid ClojureScript makes visual interactive syntax a language extension.
   - A VIsx is not just a visualization of code; editing the mini-GUI changes the underlying code representation.
   - Every VIsx also has a textual representation, so ordinary editors can still view and edit the program.
   - The authors treat VIsx as syntax inside the AST, which allows visual and textual forms to nest and compose.
3. The design prioritizes static reasoning and composability over unrestricted liveness.
   - The paper connects the approach to macro systems in Scheme, Racket, Rust, Scala 3, Julia, and Elixir.
   - Separate edit-time, compile-time, and run-time phases limit what extension code can access.
   - This phase separation helps preserve binding, hygiene, analysis, refactoring, and composability across extensions.
4. The second implementation improves adoptability by using ClojureScript, CodeMirror, and the DOM.
   - The earlier Racket prototype had to reimplement retained GUI widgets on top of an immediate-mode canvas.
   - Hybrid ClojureScript instead uses the DOM, allowing VIsx authors to reuse browser and IDE GUI infrastructure.
   - The paper says DOM reuse also makes libraries such as React, Angular, Vue.js, and Bootstrap available to extension authors.
5. The evaluations show visual syntax across concrete programming tasks.
   - Tsuro examples use visual tiles and boards to express unit tests that would be hard to read as lists of letters.
   - Red-black tree patterns and meta-extension form builders test whether VIsx can cooperate with abstraction mechanisms.
   - API protocol state machines and Catan board examples show reuse of generic visual libraries for domain-specific code.

### 2. What I’m Finding About LLM Code Style and Token Costs

1. The essay starts from a practical observation about LLM-generated boilerplate.
   - Montgomery says Claude often defaults to older Node.js-style patterns unless the runtime context is explicit.
   - The article frames output tokens as the expensive side of generation, often costing three to five times input tokens in API pricing.
   - The central claim is that models waste tokens when they reimplement problems the Web platform already solved.
2. Native Web APIs are presented as the largest lever.
   - The essay compares manual query parsing at about 140 tokens with a `URLSearchParams` approach at about 12 tokens.
   - Form handling falls from roughly 200-250 tokens of per-field state to a short `FormData` call.
   - Fetch cancellation and parallel async examples favor `AbortSignal.timeout(5000)` and `Promise.allSettled()` over ad hoc lifecycle code.
3. The argument is about correctness as much as cost.
   - Manual query parsing can mishandle malformed keys, repeated parameters, encoding, and prototype pollution.
   - Custom timer cleanup, form state tracking, and modal focus management create reliability and accessibility risk.
   - Native APIs are spec-defined and tested through platform interoperability suites instead of recreated from memory each generation.
4. Comments and formatting receive a more nuanced treatment.
   - Montgomery cites research suggesting stale comments can degrade model comprehension below a no-comment baseline.
   - The recommended comment style records constraints, invariants, and reasons rather than restating the next line of code.
   - Formatting reductions can save input tokens, but the essay treats those gains as secondary to eliminating needless output.
5. The practical prescription is to constrain model defaults early.
   - Montgomery recommends naming APIs such as `URL`, `URLSearchParams`, `FormData`, `Headers`, `Request`, `Response`, and `Promise.allSettled()` at the start of a session.
   - The essay also recommends semantic HTML such as `<dialog>`, `<details>`, and native form validation.
   - The broader professional point is that someone must know what the runtime already ships before asking the model to generate code.

### 3. How to Be a 30x AI Engineer with a Taste

1. The post argues that AI has shifted the scarce engineering skill away from typing code.
   - Bhavsar cites recent claims from AI-company engineers and leaders that code generation has crossed a major capability threshold.
   - The essay treats implementation speed as increasingly commoditized when models can produce usable code in minutes.
   - What remains valuable is problem decomposition, product judgment, architecture, reliability, user understanding, and shipping judgment.
2. Taste is defined through three complementary lenses.
   - Taste as recognition is the ability to see which of two implementations is better before fully articulating the reason.
   - Taste as compass is the internal standard that guides tradeoffs when no checklist gives the answer.
   - Taste as vision is the ability to imagine a better end state, not merely compare existing options.
3. The post maps taste across engineering value zones.
   - Quality judgment covers whether code is clean, maintainable, efficient, secure, and aligned with the codebase.
   - Product and user judgment cover which features deserve to exist and how work should feel to the people using it.
   - Architecture, communication, risk, and organizational judgment decide how systems evolve beyond one local implementation.
4. Bhavsar proposes deliberate practice instead of hoping taste appears passively.
   - The suggested path includes reverse-engineering excellent work, comparing alternatives, and writing down why creators made specific decisions.
   - A 90-day plan moves from recognition to discrimination to generative application.
   - The post recommends projects such as an AI-code evaluation framework, better open-source onboarding, team taste tests, constrained shipping, and opinionated technical writing.
5. The career advice is to position around judgment-heavy work.
   - Bhavsar warns that competing on raw coding speed is a weak strategy when AI can generate code quickly.
   - The essay recommends adjacent skills such as product thinking, design awareness, business understanding, and communication.
   - Roles such as founding engineer, tech lead, platform engineer, developer experience engineer, and staff+ engineer are presented as places where taste compounds.
