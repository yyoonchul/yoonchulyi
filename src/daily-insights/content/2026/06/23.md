## 📝 Detailed Notes

### 1. Death by a Thousand Comments

1. The post reacts to enthusiasm around engineering-team context layers.
   - Patrick Smith cites Gergely Orosz's observation that many VC-funded and bootstrapped products are trying to gather engineering context from other systems.
   - Smith accepts that teams feel a real pain point, but disputes that the missing layer is mainly a technical integration layer.
   - The post reframes the desired context layer as leadership with clear communication rather than another system that indexes all work tools.
2. The article argues that modern collaboration tools scatter one conversation across many places.
   - Smith points to comments living separately in Figma, GitHub pull requests, Slack messages, issue trackers, monitoring tools, and documents.
   - The same coworkers effectively appear in multiple tool-specific identities, forcing teammates to reconstruct intent from disconnected fragments.
   - The cognitive cost comes from having to assemble a consistent model of decisions, constraints, and status from many partial records.
3. Smith contrasts tool sprawl with older, simpler communication patterns.
   - The essay mentions older Apple and Microsoft habits around email as an example of blunt but clear written coordination.
   - The point is not that email is ideal, but that direct communication can be more useful than polished executive summaries, stale diagrams, or fragmented updates.
   - Single sign-on and SaaS adoption made it easy to create many small workspaces, but those workspaces did not necessarily improve alignment.
4. AI search is presented as a tempting but incomplete response to organizational confusion.
   - Smith understands why teams want a system that can search every stray word and produce a coherent timeline.
   - The post argues that the underlying problem is not only retrieval, but the organization's habit of denormalizing decisions across many private clouds.
   - More glue between systems may preserve the same complexity instead of reducing the need for teammates to interpret and decide together.
5. The essay's prescription is more active leadership and shared context formation.
   - Smith says leaders should shape communication, consume context, inspect it, and direct action.
   - The post treats human agency as central, questioning why probabilistic software loops get called agents while people become passive around information overload.
   - The final argument is that teams need direct conversation and clear decisions, not just more tools that summarize conversations after alignment has already failed.

### 2. JSON-LD Explained for Personal Websites

1. The post introduces JSON-LD as structured metadata for personal sites.
   - Ethan Hawksley describes JSON-LD as JSON Linked Data placed in a page's `<head>` with the `application/ld+json` MIME type.
   - The browser does not execute that script as JavaScript, while crawlers such as Googlebot can parse it for semantic page information.
   - The guide uses `https://schema.org` as the context and recommends thinking of the document as a labeled directed graph under `@graph`.
2. Hawksley explains why stable node identifiers matter.
   - Each node has an `@type`, an `@id`, and properties that describe the entity represented by that node.
   - A URL plus a hash fragment, such as `#website` or `#person`, gives crawlers a stable identifier for merging information across pages.
   - The article notes that search crawlers can merge properties across pages, while single-page scrapers may only see the current page's graph.
3. The guide covers site-level and person-level nodes first.
   - `WebSite` describes the overall site and can be fully detailed on the root page while remaining slim on other pages.
   - `WebPage` describes the current physical HTML page and can link to the parent `WebSite`, breadcrumbs, language, and primary image.
   - `Person` is recommended on all pages for personal websites because it helps identify the site owner and connect profiles, images, names, languages, and topics.
4. The post maps common personal-site pages and projects to Schema.org types.
   - `ProfilePage` works for a home or about page, with `mainEntity` linking the page to the person it describes.
   - `SoftwareApplication` is useful for showcased projects and can include deployed URL, category, operating system, creator, repository-like `sameAs` links, and free `Offer` metadata.
   - `CollectionPage`, `Blog`, and `BlogPosting` help distinguish list pages, blog indexes, and individual posts instead of treating every URL as a generic page.
5. The article focuses on practical SEO and crawler outcomes rather than semantic-web theory.
   - Breadcrumb metadata can influence how search results show a page path, especially when URL paths are long.
   - Blog post metadata can include headline, description, section, keywords, language, dates, author, publisher, image, and license.
   - The conclusion is that even static personal sites can gain useful structure by adding at least `WebSite`, `ProfilePage`, and `Person` to the root page, then richer post and project nodes where relevant.

### 2. JSON-LD Explained for Personal Websites

1. 글은 JSON-LD를 개인 사이트를 위한 구조화 metadata로 소개한다.
   - Ethan Hawksley는 JSON-LD를 `application/ld+json` MIME type으로 페이지의 `<head>`에 넣는 JSON Linked Data라고 설명한다.
   - browser는 그 script를 JavaScript로 실행하지 않지만, Googlebot 같은 crawler는 의미론적 페이지 정보를 얻기 위해 파싱할 수 있다.
   - 가이드는 `https://schema.org`를 context로 사용하며, 문서를 `@graph` 아래의 labeled directed graph로 생각하라고 권한다.
2. Hawksley는 안정적인 node identifier가 중요한 이유를 설명한다.
   - 각 node에는 해당 node가 나타내는 entity를 설명하는 `@type`, `@id`, property가 있다.
   - `#website`나 `#person` 같은 hash fragment가 붙은 URL은 crawler가 여러 페이지의 정보를 병합할 수 있는 안정적인 identifier를 제공한다.
   - 글은 search crawler는 여러 페이지의 property를 병합할 수 있지만, single-page scraper는 현재 페이지의 graph만 볼 수 있다고 지적한다.
3. 가이드는 먼저 site-level과 person-level node를 다룬다.
   - `WebSite`는 전체 사이트를 설명하며 root page에는 자세히 넣고 다른 페이지에는 더 간략하게 둘 수 있다.
   - `WebPage`는 현재의 물리적 HTML page를 설명하며 상위 `WebSite`, breadcrumb, language, primary image에 연결될 수 있다.
   - `Person`은 사이트 소유자를 식별하고 profile, image, name, language, topic을 연결하는 데 도움이 되므로 개인 웹사이트의 모든 페이지에 권장된다.
4. 글은 일반적인 개인 사이트 페이지와 프로젝트를 Schema.org type에 매핑한다.
   - `ProfilePage`는 home 또는 about page에 맞으며, `mainEntity`로 페이지가 설명하는 사람과 연결한다.
   - `SoftwareApplication`은 공개한 프로젝트에 유용하며 배포 URL, category, operating system, creator, repository 같은 `sameAs` link, 무료 `Offer` metadata를 포함할 수 있다.
   - `CollectionPage`, `Blog`, `BlogPosting`은 모든 URL을 generic page로 취급하지 않고 list page, blog index, 개별 post를 구분하는 데 도움을 준다.
5. 기사는 semantic web 이론보다 실용적인 SEO와 crawler 결과에 초점을 맞춘다.
   - Breadcrumb metadata는 특히 URL path가 길 때 검색 결과에서 페이지 경로가 표시되는 방식에 영향을 줄 수 있다.
   - Blog post metadata에는 headline, description, section, keyword, language, date, author, publisher, image, license가 포함될 수 있다.
   - 결론은 static personal site라도 root page에 최소한 `WebSite`, `ProfilePage`, `Person`을 추가하고, 필요할 때 더 풍부한 post와 project node를 넣으면 유용한 구조를 얻을 수 있다는 것이다.
