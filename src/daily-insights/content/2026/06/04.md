## 📝 Detailed Notes

### 1. Three ways founders are building in public

1. The article says the old "build in public" playbook has lost force.
   - Earlier versions centered on Product Hunt launches, social amplification, and hoping customers would appear after a coordinated launch push.
   - A later indie-hacker version leaned into revenue dashboards and selling to an audience watching the founder's progress.
   - a16z speedrun frames the current backlash against hustle culture as a reason founders need a more credible public posture.
   - The replacement thesis is "publicly excellent": make the company itself the content rather than treating marketing as a separate performance.
2. The first pattern is to ship visible frontier capabilities.
   - The example is Tom Krcha at Pencil, an AI design tool described as living where users code.
   - Pencil crossed 100,000 users in May and introduced Swarm mode, where six AI design agents work on one canvas in parallel.
   - The launch works as public building because the demo shows a capability that would not have been feasible a year earlier.
   - Krcha's credibility matters: the article notes that Krcha previously sold companies to Miro and Google, so the update reads like an operator reporting from the edge.
3. The second pattern is to make the company's strategic bet public.
   - The article uses Gainsight CEO Chuck Ganapathi as the example of turning a business-model transition into an open storyline.
   - Gainsight, described as having hundreds of millions in ARR, is publicly moving toward AI-native services over traditional SaaS.
   - Its agent product Atlas is being treated as its own P&L, making the AI-native services bet measurable and visible.
   - The article argues that the public nature of the bet gives future updates more narrative weight because the stakes are already clear.
4. The third pattern is to make unconventional hiring part of the product signal.
   - The example is Jean-Michel Lemieux joining Spellbook, a contracts startup that raised a $50M Series B from Khosla.
   - Lemieux previously served as Shopify CTO and helped scale engineering from about 50 people to roughly 4,000, after earlier engineering leadership at Atlassian.
   - Instead of joining Spellbook as CTO, Lemieux joined as an Executive IC, a hands-on role without the conventional title leverage or reporting structure.
   - The article frames that move as a public proof point: the company can attract unusually strong talent into an unusual seat because the mission is credible.
5. The underlying rule is substance before spectacle.
   - a16z speedrun argues that founders earn the right to build in public only when the work itself is worth attention.
   - The article contrasts meaningful public operating signals with generic posts about working hard.
   - Shipping the frontier, betting the business, and landing a surprising senior hire all show how the company works differently.
   - The practical lesson is that founders should expose real strategic motion, not just convert ordinary operations into performative updates.

### 2. A harness for every task: dynamic workflows in Claude Code

1. Dynamic workflows let Claude Code create a task-specific harness.
   - Thariq says Claude can now write a custom JavaScript workflow on the fly for the task at hand.
   - The default Claude Code harness is optimized for coding, but many research, security, review, and coordination tasks also resemble coding tasks.
   - Workflows let Claude create and reuse harnesses that coordinate subagents more natively inside Claude Code.
   - The article cautions that best practices are still developing and that workflows can consume more tokens.
2. The feature targets failure modes that appear in long or structured tasks.
   - A single context window must both plan and execute, which works well for many coding tasks but can break down over long runs.
   - Thariq names agentic laziness, self-preferential bias, and goal drift as common problems.
   - Agentic laziness means stopping after partial progress; self-preferential bias means over-trusting the model's own results; goal drift means losing original requirements after many turns or compaction.
   - Workflows counter these issues by giving separate Claude instances focused context windows, isolated goals, and verifier roles.
3. Workflows are dynamic rather than static orchestration scripts.
   - Static workflows built with the Claude Agent SDK or `claude -p` need to be generic enough for many edge cases.
   - With Claude Opus 4.8 and dynamic workflows, Claude can write a harness tailored to the current task.
   - Workflow code can decide which models agents use and whether subagents run in separate worktrees.
   - If a workflow is interrupted, resuming the session can allow the workflow to continue from the prior state.
4. The article lists reusable workflow patterns.
   - Classify-and-act uses a classifier agent to route tasks or decide final output categories.
   - Fan-out-and-synthesize splits work into smaller independent steps, then waits at a synthesis barrier to merge structured outputs.
   - Adversarial verification runs separate agents to test outputs against rubrics, while generate-and-filter creates many candidates and keeps the strongest ones.
   - Tournament workflows let agents compete on the same task, with judging agents comparing results until a winner emerges.
5. The use cases extend beyond ordinary coding.
   - For migrations and refactors, workflows can split callsites, modules, and failing tests across worktrees, then review and merge fixes.
   - For deep research, workflows can fan out web searches, fetch sources, verify claims, and synthesize cited reports.
   - For sorting large qualitative datasets, tournament or bucket-ranking workflows avoid asking one context window to score 1,000+ rows at once.
   - For root-cause analysis, workflows can generate hypotheses from disjoint evidence sources and use verifier or refuter agents to test them.
6. Thariq gives practical guidance on when to use workflows.
   - Users can ask Claude to create a workflow directly, or use a trigger word such as "ultracode" to encourage Claude Code to build one.
   - Workflows are most useful when a task benefits from parallelism, independent judgment, adversarial review, repeated loops, or a formal rubric.
   - They are not necessary for every coding task, and a routine implementation probably does not need a panel of multiple reviewers.
   - The article recommends explicit prompting, pairing repeatable workflows with `/goal` and `/loop`, setting token budgets, and saving reusable workflows into `~/.claude/workflows` or a skill.

### 2. A harness for every task: dynamic workflows in Claude Code

1. Dynamic workflows는 Claude Code가 작업별 harness를 만들게 한다.
   - Thariq는 Claude가 현재 task에 맞춘 custom JavaScript workflow를 즉석에서 작성할 수 있다고 설명한다.
   - 기본 Claude Code harness는 coding에 최적화돼 있지만, research, security, review, coordination task 중 상당수도 coding task와 닮아 있다.
   - Workflow는 Claude가 subagent를 조율하는 harness를 만들고 재사용해 Claude Code 안에서 이런 문제를 더 자연스럽게 풀도록 한다.
   - 글은 best practice가 아직 발전 중이며 workflow가 더 많은 token을 사용할 수 있다고 주의시킨다.
2. 이 기능은 길거나 구조화된 작업에서 나타나는 failure mode를 겨냥한다.
   - 단일 맥락 창은 planning과 execution을 동시에 해야 하며, 많은 coding task에는 효과적이지만 장기 실행 작업에서는 무너질 수 있다.
   - Thariq는 agentic laziness, self-preferential bias, goal drift를 흔한 문제로 든다.
   - Agentic laziness는 일부만 진행하고 멈추는 현상이고, self-preferential bias는 model 자신의 결과를 과신하는 현상이며, goal drift는 많은 turn이나 compaction 뒤 원래 요구사항을 잃는 현상이다.
   - Workflow는 별도 Claude instance에 focused context window, isolated goal, verifier role을 부여해 이런 문제를 줄인다.
3. Workflow는 정적인 orchestration script가 아니라 dynamic하다.
   - Claude Agent SDK나 `claude -p`로 만든 static workflow는 많은 edge case를 감당할 만큼 generic해야 한다.
   - Claude Opus 4.8과 dynamic workflows에서는 Claude가 현재 task에 맞춘 harness를 작성할 수 있다.
   - Workflow code는 agent가 어떤 model을 쓸지, subagent가 별도 worktree에서 실행될지를 결정할 수 있다.
   - Workflow가 중단되더라도 session을 resume하면 이전 상태에서 계속 진행할 수 있다.
4. 글은 재사용 가능한 workflow pattern을 나열한다.
   - Classify-and-act는 classifier agent로 task를 route하거나 최종 output category를 결정한다.
   - Fan-out-and-synthesize는 작업을 독립적인 작은 단계로 나눈 뒤 synthesis barrier에서 기다렸다가 structured output을 병합한다.
   - Adversarial verification은 별도 agent가 rubric에 따라 output을 검증하게 하고, generate-and-filter는 많은 candidate를 만든 뒤 가장 강한 것만 남긴다.
   - Tournament workflow는 agent들이 같은 task에서 경쟁하게 하고, judging agent가 결과를 비교해 winner를 고른다.
5. 사용 사례는 일반 coding을 넘어선다.
   - Migration과 refactor에서는 workflow가 callsite, module, failing test를 worktree별로 나눠 fix한 뒤 review하고 merge할 수 있다.
   - Deep research에서는 workflow가 web search를 fan out하고, source를 fetch하며, claim을 verify하고, citation이 있는 report를 synthesize할 수 있다.
   - 대규모 qualitative dataset sorting에서는 tournament나 bucket-ranking workflow가 단일 context window에 1,000개 이상의 row를 한 번에 scoring하게 하는 문제를 피한다.
   - Root-cause analysis에서는 workflow가 서로 분리된 evidence source에서 hypothesis를 만들고 verifier나 refuter agent로 test할 수 있다.
6. Thariq는 workflow 사용 시점에 대한 실용적 지침을 제시한다.
   - 사용자는 Claude에게 workflow 생성을 직접 요청하거나, "ultracode" 같은 trigger word를 사용해 Claude Code가 workflow를 만들도록 유도할 수 있다.
   - Workflow는 task가 parallelism, independent judgment, adversarial review, repeated loop, formal rubric에서 이득을 볼 때 가장 유용하다.
   - 모든 coding task에 필요한 것은 아니며, 일상적인 구현 작업은 여러 reviewer panel을 필요로 하지 않을 가능성이 크다.
   - 글은 구체적 prompting, 반복 가능한 workflow와 `/goal` 및 `/loop`의 결합, token budget 설정, 재사용 가능한 workflow를 `~/.claude/workflows`나 skill에 저장하는 방식을 권한다.
