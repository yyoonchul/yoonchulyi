### 3. Codex-maxxing for long-running work

1. OpenAI's guide focuses on using Codex for larger, longer-running tasks.
   - The page introduces a 33-page PDF guide for getting more value from Codex in both ChatGPT and local CLI workflows.
   - The guide targets tasks that require investigation, implementation, testing, and iteration rather than single-turn answers.
   - OpenAI frames these workflows as closer to managing an agent process than asking isolated coding questions.
2. The central recommendation is to give Codex durable goals and context.
   - OpenAI recommends defining the desired outcome, relevant constraints, validation steps, and stop conditions upfront.
   - Files such as `AGENTS.md`, planning notes, and progress logs help Codex resume work without repeatedly reconstructing project rules.
   - The guide treats good context as an asset that should survive across sessions, branches, and execution environments.
3. The guide separates work across ChatGPT, the Codex CLI, and async execution.
   - ChatGPT is useful for planning, reviewing, and delegating work at a higher level.
   - The Codex CLI is useful when Codex needs direct repository access, local commands, and iterative code edits.
   - Async Codex tasks are positioned for work that can run in the background and return with a branch, summary, or reviewable change set.
4. Advanced usage depends on checkpoints and review loops.
   - OpenAI recommends asking Codex to make measurable progress, run tests, and report what changed instead of simply claiming completion.
   - Long work benefits from explicit checkpoints such as saved plans, issue lists, test results, and known remaining risks.
   - Human review still matters because Codex can accelerate execution, but the user remains responsible for acceptance and deployment decisions.
5. The broader shift is from prompting to operating an agent workflow.
   - The guide encourages users to describe outcomes and evaluation criteria rather than micromanaging every edit.
   - It also emphasizes environment setup, tool access, and repository conventions because these determine how effectively Codex can act.
   - The practical message is that Codex performs best when treated as a collaborator with memory, tools, and verification routines.

### 4. Loop Engineering Clearly Explained

1. Akshay defines an agent as a loop around a model, tools, and context.
   - The loop starts when a model reads context, requests tool calls, receives tool results, and repeats until no more tool calls are requested.
   - The article argues that this basic loop is not the hard part because serious agent frameworks converge on similar control flow.
   - The engineering challenge is everything around the loop that makes repeated autonomous work useful instead of brittle.
2. The article says agent engineering has moved beyond prompt wording.
   - Prompt engineering covers the words sent to a model, while context engineering controls what the model sees.
   - Harness engineering adds the surrounding code that runs tools, tracks state, and handles failures.
   - Loop engineering coordinates the autonomous cycle that keeps pushing toward a goal across multiple steps.
3. Stopping correctly is presented as the first hard problem.
   - An agent ending its turn is not the same as the task being finished.
   - Good loops need max-iteration caps, time and cost limits, no-progress detection, and explicit completion checks.
   - The article's standard for "done" is an external proof such as passing tests, not the agent's own confidence.
4. Context and tools determine whether the loop degrades or improves.
   - Long-running agents can suffer from context rot as stale outputs, dead ends, and unnecessary reasoning accumulate.
   - The proposed defenses are compaction, offloading large outputs, and isolating messy subtasks in sub-agents.
   - Tool sets should be focused, writes should be safe to repeat, and error messages should tell an agent what to try next.
5. A reliable loop needs a critic that can reject weak work.
   - Akshay argues that an autonomous agent can agree with itself unless the system includes tests, type checks, real errors, or another reviewer.
   - The maker and checker roles should be separated so the worker is not the only judge of success.
   - The final takeaway is that the model may become more interchangeable, while the loop around the model becomes the main engineering advantage.

5. 더 넓은 변화는 prompting에서 agent workflow 운영으로 이동하는 것이다.
   - 가이드는 사용자가 모든 edit을 micromanage하기보다 outcome과 evaluation criteria를 설명하도록 권한다.
   - 또한 environment setup, tool access, repository convention을 강조하는데, 이것들이 Codex가 얼마나 효과적으로 행동할 수 있는지를 결정하기 때문이다.
   - 실용적 메시지는 Codex를 memory, tool, verification routine을 가진 collaborator로 대할 때 가장 잘 작동한다는 것이다.

### 4. Loop Engineering Clearly Explained

1. Akshay는 agent를 model, tool, context 주변의 loop로 정의한다.
   - 이 loop는 model이 context를 읽고, tool call을 요청하고, tool result를 받은 뒤, 더 이상 tool call이 요청되지 않을 때까지 반복하면서 시작된다.
   - 글은 serious agent framework가 비슷한 control flow로 수렴하므로 이 기본 loop 자체가 어려운 부분은 아니라고 주장한다.
   - engineering challenge는 반복적 자율 작업을 brittle하지 않고 유용하게 만드는 loop 주변의 모든 것에 있다.
2. 글은 agent engineering이 prompt 문구를 넘어섰다고 말한다.
   - Prompt engineering은 model에 보내는 단어를 다루고, context engineering은 model이 보는 것을 통제한다.
   - Harness engineering은 tool을 실행하고, state를 추적하고, failure를 처리하는 주변 code를 더한다.
   - Loop engineering은 여러 step에 걸쳐 goal을 향해 계속 밀고 가는 autonomous cycle을 조정한다.
3. 올바르게 멈추는 것이 첫 번째 어려운 문제로 제시된다.
   - Agent가 turn을 끝내는 것은 task가 끝난 것과 같지 않다.
   - 좋은 loop에는 max-iteration cap, 시간과 비용 limit, no-progress detection, 명시적 completion check가 필요하다.
   - 글에서 "done"의 기준은 agent 자신의 confidence가 아니라 test 통과 같은 외부 proof다.
4. Context와 tool은 loop가 나빠지는지 개선되는지를 결정한다.
   - 오래 실행되는 agent는 stale output, dead end, 불필요한 reasoning이 쌓이면서 context rot을 겪을 수 있다.
   - 제안된 방어책은 compaction, 큰 output offloading, messy subtask를 sub-agent에 격리하는 것이다.
   - Tool set은 focused해야 하고, write 작업은 반복해도 안전해야 하며, error message는 agent에게 다음에 무엇을 시도할지 알려줘야 한다.
5. 신뢰 가능한 loop에는 약한 작업을 거부할 수 있는 critic이 필요하다.
   - Akshay는 system에 test, type check, 실제 error, 또는 다른 reviewer가 없으면 autonomous agent가 자기 자신에게 동의할 수 있다고 주장한다.
   - Worker가 성공의 유일한 judge가 되지 않도록 maker와 checker 역할을 분리해야 한다.
   - 최종 결론은 model이 더 상호교환 가능해질 수 있고, model 주변의 loop가 핵심 engineering advantage가 된다는 것이다.
