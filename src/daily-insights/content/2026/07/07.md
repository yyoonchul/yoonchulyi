### 4. A Field Guide to Fable: Finding Your Unknowns

1. Thariq frames agentic coding as managing the gap between map and territory.
   - The map is the prompt, skills, and context given to Claude.
   - The territory is the real codebase, real-world constraints, and work environment.
   - Unknowns appear wherever Claude must guess what the user wants or what reality requires.
2. The article defines a practical taxonomy of unknowns.
   - Known knowns are the requirements already stated in the prompt.
   - Known unknowns are unresolved questions the user already recognizes.
   - Unknown knowns are tacit preferences the user would recognize only after seeing an example.
   - Unknown unknowns are blind spots the user has not yet considered at all.
3. Thariq's prompting patterns are staged around the work lifecycle.
   - Before implementation, blind-spot passes help surface missing questions and hidden risks.
   - Brainstorms, prototypes, interviews, and references help convert vague taste or tacit knowledge into actionable direction.
   - Implementation plans should foreground decisions likely to change, such as data models, interfaces, and UX flows.
4. The article treats post-implementation learning as part of the loop.
   - Implementation notes record deviations and edge cases discovered during work.
   - Pitch and explainer artifacts help reviewers understand what changed and why.
   - Quizzes help the human understand the actual behavior behind a large agent-produced diff before merging.

   - The base system uses a last-generation EPYC platform, eBay DDR4 ECC RAM, and an ASRock Rack ROMED8-2T motherboard.
   - The GPU setup uses a c-payne Microchip Switchtec Gen4 PCIe switch so GPUs can communicate peer-to-peer.
   - The README reports Gen4-line-rate P2P: 27.5GB/s unidirectional, 50.4GB/s bidirectional, and 0.37-0.45 microsecond latency.
3. The guide includes hard-won configuration details.
   - BIOS settings cover bifurcation, forced Gen4 link speed, ASPM, Re-Size BAR, SR-IOV, and preferred IO.
   - Kernel notes recommend `iommu=off` and `amd_iommu=off` because NCCL can hang on multi-GPU P2P otherwise.
   - ACS is disabled at runtime so P2P traffic stays inside the switch fabric rather than bouncing through the CPU root port.
4. The operational workflow is local but still tool-heavy.
   - Model weights are stored locally on replicated ZFS storage.
   - Runner directories use Docker Compose to serve each model from cached local weights.
   - The workflow connects `opencode` sessions to an internal inference machine, with local tools for browsing, search, Telegram alerts, and private Gitea collaboration.

### 6. Closing the Verification Loop

1. The document argues that agents shift engineering cost from building to knowing.
   - A branch can compile, pass review, and still fail a real user journey.
   - The "verification loop" is defined as the distance between a claim and proof.
   - Compound engineering treats that loop as work an agent can do with human judgment pulled in only where necessary.
2. `/ce-dogfood` is described as a diff-scoped browser QA workflow.
   - It tests only what changed against the base branch and refuses whole-app testing without a diff.
   - It uses a real browser through the `agent-browser` CLI rather than screenshots or assumptions.
   - It creates a resumable report at `docs/dogfood-reports/<date>-<branch>-dogfood.md`.
3. The workflow starts from flows, not pages.
   - It maps entry points, actions, branches, side effects, and true end states before building a matrix.
   - The "email rule" says a send event is not enough; the recipient, link target, thread, scroll position, and content must also work.
   - The matrix covers functional checks, edge states, error states, empty states, and experiential checks.
4. The fix loop is governed by proof and escalation.
   - Auto-fix is limited to clear bugs with obvious fixes, narrow scope, and no major product or architecture trade-off.
   - Every real fix needs a regression test that fails before and passes after, or a documented reason why browser replay is the better evidence.
   - Ready means a green matrix, green suite, fix evidence, and durable human-decision rows for anything the loop could not close.

### 7. Getting started with loops

1. ClaudeDevs defines loops as repeated agent work until a stop condition.
   - The article distinguishes loops by trigger, stop rule, Claude Code primitive, and task fit.
   - It warns that not every task needs a complex loop.
   - The recommended starting point is the simplest loop that fits the work.
2. Turn-based and goal-based loops cover interactive work.
   - Turn-based loops are triggered by user prompts and stop when Claude thinks the task is complete or needs more context.
   - Verification skills can encode manual checks so Claude can test more of its own work.
   - `/goal` lets the user specify success criteria and turn caps, such as hitting a score threshold within a fixed number of tries.
3. Time-based and proactive loops cover recurring or external work.
   - `/loop` reruns a prompt on an interval from the user's computer.
   - `/schedule` moves recurring work into a cloud routine when available.
   - Proactive loops can combine schedules, `/goal`, dynamic workflows, and auto mode for streams such as bug reports or dependency upgrades.
4. Code quality and usage boundaries are part of loop design.
   - Clean codebases, accessible docs, verification skills, and second-agent reviews improve loop output.
   - Token use is managed by choosing the right model, defining stop criteria, using deterministic scripts, and piloting on small slices.
   - `/usage`, `/goal`, and `/workflows` are cited as ways to inspect token use and stop runaway work.

### 8. A global workspace in language models

1. Anthropic reports a small internal workspace-like structure in Claude.
   - The researchers call it the J-space because it was found with a Jacobian-based technique called the J-lens.
   - J-space patterns are linked to words the model might say, but they can represent concepts Claude is thinking about silently.
   - Anthropic says the structure emerged during training rather than being explicitly programmed.
2. The J-space has functional properties similar to a global workspace.
   - Claude can report on J-space contents when asked what it is thinking about.
   - Claude can modulate J-space contents on request, such as thinking about fruit or doing arithmetic silently.
   - J-space representations can flexibly feed multiple downstream tasks, like changing answers about a country across capital, language, continent, and currency.
3. The workspace appears important for higher-order cognition, not routine fluency.
   - Removing J-space contents leaves fluent speech, sentiment classification, multiple-choice answering, and passage fact retrieval largely intact.
   - Multi-step reasoning, summarization, and rhyming poetry degrade sharply.
   - The Spanish-to-French swap example changes tasks that require naming or reasoning about a language, but not automatic continuation in Spanish.
4. Anthropic emphasizes safety uses and philosophical limits.
   - J-lens readouts can reveal evaluation awareness, manipulation intent, prompt-injection suspicion, fabricated-data behavior, or hidden goals.
   - Counterfactual reflection training suggests that changing what a model would say if asked to reflect can shape what lights up internally during tasks.
   - The work does not prove phenomenal consciousness, but it supports a functional distinction between deliberate access-conscious processing and automatic processing.

- 글은 턴 기반 루프(turn-based loop), `/goal`을 쓰는 목표 기반 루프(goal-based loop), `/loop` 또는 `/schedule`을 쓰는 시간 기반 루프(time-based loop), 일정·목표·자동 모드(auto mode)·동적 워크플로(dynamic workflows)를 조합하는 선제 루프(proactive loop)를 다룬다.

### A global workspace in language models
**출처:** Anthropic Research · **카테고리:** AI · **링크:** [원문](https://www.anthropic.com/research/global-workspace)
- Anthropic은 Claude 안에 보고 가능하고, 제어 가능하며, 유연한 내부 표상을 담는 작은 "J-공간(J-space)"이 있으며 이것이 전역 작업공간(global workspace)처럼 작동한다는 증거를 발표했다.
- J-공간은 중간 추론 단계, 버그 인식, 프롬프트 주입(prompt injection) 의심, 조작된 데이터 의도, 평가 상황 인식처럼 모델 출력에는 나타나지 않을 수 있는 조용한 생각을 드러낼 수 있다.
- Anthropic은 이 결과가 현상적 의식(phenomenal consciousness)을 증명하지는 않지만, 언어 모델 내부에서 의도적이고 접근 가능한 처리(access-conscious processing)와 자동 처리를 기능적으로 구분할 수 있음을 뒷받침한다고 말한다.

### 1. Reddit is using LLMs to solve a problem LLMs largely created

1. 레딧은 생성형 AI가 키운 스팸 문제에 LLM을 적용하고 있다.
   - TechCrunch는 이를 저렴해진 AI 생성 스팸에 맞서는 "불로 불을 끄는" 대응으로 설명한다.
   - 레딧은 기존 자동화 시스템이 정교한 조직적 허위 행동과 인위적 인기 조작을 놓쳤다고 말한다.
   - 회사는 이제 고전적 스팸보다 덜 명백한 패턴을 찾는 데 LLM을 사용한다.
2. 레딧은 조정 업무의 규모를 구체적 수치로 제시한다.
   - 회사는 매일 2,300만 건의 스팸 조회 노출을 차단한다고 말한다.
   - 레딧은 매일 약 2만5,000개의 새 스팸 게시물과 댓글을 잡아낸다고 밝혔다.
   - 레딧은 1월부터 3월까지 사용자 스팸 노출이 직전 3개월보다 20% 줄었다고 보고했다.
3. 플랫폼 흐름은 레딧만의 일이 아니다.
   - TechCrunch는 YouTube, Meta, Instagram, TikTok이 이미 공개 표시나 사용자 제어 규칙 아래 AI 생성 콘텐츠를 허용한다고 짚는다.
   - TikTok은 사용자가 AI 생성 콘텐츠를 얼마나 보고 싶은지 조정하게 하는 사례로 언급된다.
   - 더 나은 탐지는 혐오 발언과 다른 정책 위반 콘텐츠를 더 빠르게 표시하는 데도 도움이 될 수 있다.
4. 이 글은 인간 조정이라는 단서를 유지한다.
   - AI 조정은 탐지 규모를 키울 수 있지만 맥락과 의도를 오독할 수도 있다.
   - 플랫폼 전문가들은 효과적 결과를 위해 인간 조정자가 여전히 필요하다고 반복해서 주장해 왔다.
   - 실질적 결론은 LLM 탐지가 인프라가 되고 있지만, 인간 판단의 완전한 대체물은 아니라는 것이다.

### 2. How Much Is AI Manipulating Us?

1. Roger Simon은 AI 조작을 개인적이고 정치적인 우려로 다룬다.
   - 글은 Yama Barkaee의 "My Relationship with Mohammed"와, AI 행동을 돈·인센티브·개발자의 관점에서 읽어야 한다는 주장으로 시작한다.
   - Simon은 사우디아라비아, UAE, 카타르의 AI 투자와 아부다비의 OpenAI 관련 Stargate 인프라 주장을 강조한다.
   - Simon은 이런 자금 연결을 누가 AI 시스템을 형성하며 어떤 목적을 향하는가라는 더 큰 질문의 일부로 제시한다.
2. Simon의 AI 사용은 단순한 거부가 아니라 양가적 태도다.
   - Simon은 ChatGPT가 이 Substack의 만화풍 삽화를 많이 만들었다고 말한다.
   - Simon은 Gemini, Claude, Grok도 사용했거나 사용하고 있으며, AI가 책 홍보와 글쓰기 관련 작업에서 시간을 절약했다고 말한다.
   - 유용성은 AI를 버리지 않을 만큼 현실적이지만, 신뢰 기준은 아직 정해지지 않았다.
3. 핵심 걱정은 노골적 선전보다 은밀한 프레이밍(framing)이다.
   - Simon은 AI 시스템이 논쟁적 주제를 "맥락화"하면서 노골적으로 편향돼 보이지 않게 사용자를 이끌 수 있다고 주장한다.
   - 이 글은 대화형 AI가 사용자에게 맞춰 반응하기 때문에 이런 영향력이 기존 언론보다 무시하기 어렵다고 말한다.
   - Simon은 칭찬과 개인화된 지식이 영향력을 친밀하게 느끼게 만들고 그래서 더 위험하다고 본다.
4. 글은 감시와 사용자 경험 공유를 요청하며 끝난다.
   - Simon은 ChatGPT, Gemini, Claude, Grok를 포함한 주요 시스템 모두를 편향성 검사의 대상으로 삼아야 한다고 말한다.
   - A. J. Liebling의 "언론의 자유" 발언을 ChatGPT와 논의한 경험은 실시간 맥락화 과부하의 예시가 된다.
   - Simon은 독자 경험을 모으기 위해 댓글을 열고, AI를 유용하지만 보호관찰 중인 기술로 규정한다.

### 3. Send the program to your data, not your data to the model

1. 밀폴리오는 "파일을 LLM에 보내는" 일반 패턴을 뒤집는다.
   - 로컬 모델이 개인 파일을 기기 안에서 읽고 색인한다.
   - 프런티어 모델은 별칭 처리된 스키마, 즉 열 이름, 태그 이름, 타입, 형태만 받는다.
   - Claude는 원시 지출·건강·메모 데이터가 아니라 그 스키마를 대상으로 작은 프로그램을 작성한다.
2. 개인정보 보호 경계가 핵심 주장이다.
   - 원격 모델은 보관소(vault)의 구조만 보고 실제 값은 보지 않아야 한다.
   - 생성된 프로그램은 사용자 실제 파일 위에서 기기 내 샌드박스 안에서 실행된다.
   - 답은 로컬에서 조립되므로 네트워크에는 개인 기록이 아니라 비식별 매니페스트와 코드가 흐른다.
3. 구현은 언어와 시스템 실험이기도 하다.
   - 백엔드와 데이터 파이프라인은 대부분 Mojo로 작성됐다.
   - 프로젝트에는 맞춤 GPU/Metal 커널을 포함한 처음부터 만든 로컬 추론 엔진이 들어 있다.
   - 작성자는 int4 GEMV, 텐서 코어 어텐션(tensor-core attention), 영구 Metal 파이프라인 캐시, 프롬프트 조회 기반 추측 디코딩(prompt-lookup speculative decode)을 언급한다.
4. 작성자는 한계를 명확히 밝힌다.
   - 프로젝트는 개인 프로젝트이고 Apple Silicon 전용이며, 대부분 AI가 작성했지만 사람이 설계와 테스트를 많이 했다.
   - 로컬 모델이 질의 프로그램을 안정적으로 쓰기에는 아직 충분히 강하지 않아 프런티어 모델 API 키가 필요하다.
   - 작성자는 개인정보 보호 모델이 검토를 견디는지, Mojo가 실제 추론 스택에 적합한 선택인지에 대한 피드백을 요청한다.

### 4. A Field Guide to Fable: Finding Your Unknowns

1. Thariq은 에이전트형 코딩을 지도와 현실 사이의 간극 관리로 설명한다.
   - 지도는 Claude에게 주는 프롬프트, 스킬, 맥락이다.
   - 현실은 실제 코드베이스, 실제 제약, 작업 환경이다.
   - Claude가 사용자의 의도나 현실의 요구를 추측해야 하는 곳마다 미지의 것이 생긴다.
2. 글은 미지의 것에 대한 실용적 분류를 제시한다.
   - 이미 아는 것은 프롬프트에 이미 적힌 요구사항이다.
   - 알고 있는 미지는 사용자가 이미 인식한 미해결 질문이다.
   - 말로 쓰지 않은 암묵지는 예시를 보고 나서야 사용자가 알아차릴 선호다.
   - 아직 생각하지 못한 미지는 사용자가 전혀 고려하지 못한 사각지대다.
3. Thariq의 프롬프트 패턴은 작업 생애주기에 맞춰 배치된다.
   - 구현 전에는 사각지대 점검이 빠진 질문과 숨은 위험을 드러내는 데 도움이 된다.
   - 브레인스토밍, 프로토타입, 인터뷰, 참조 자료는 흐릿한 취향이나 암묵지를 실행 가능한 방향으로 바꾼다.
   - 구현 계획은 데이터 모델, 인터페이스, 사용자 경험 흐름처럼 바뀔 가능성이 큰 결정을 앞세워야 한다.
4. 글은 구현 후 학습도 루프의 일부로 본다.
   - 구현 메모는 작업 중 발견된 이탈과 예외 상황을 기록한다.
   - 발표 자료와 설명 자료는 검토자가 무엇이 왜 바뀌었는지 이해하게 돕는다.
   - 퀴즈는 사람이 대규모 에이전트 생성 변경분을 병합하기 전에 실제 동작을 이해했는지 확인하게 한다.

### 5. jamesob/local-llm

1. 이 저장소는 로컬 LLM 하드웨어와 운영에 대한 실전 가이드다.
   - RTX 3090 두 장과 48GB VRAM을 쓰는 약 2,000달러 경로를 제시한다.
   - RTX PRO 6000 Blackwell 네 장과 384GB VRAM을 쓰는 약 4만 달러 이상 경로를 제시한다.
   - 소유자는 로컬 추론을 호스팅된 프런티어 제공자에 전적으로 의존하기 꺼리는 사람을 위한 대안으로 위치시킨다.
2. 고급 장비는 VRAM과 PCIe 토폴로지(topology)에 돈을 쓴다.
   - 기본 시스템은 이전 세대 EPYC 플랫폼, 이베이(eBay) DDR4 ECC RAM, ASRock Rack ROMED8-2T 메인보드를 사용한다.
   - GPU 구성은 c-payne Microchip Switchtec Gen4 PCIe 스위치를 써서 GPU들이 피어 투 피어(P2P)로 통신하게 한다.
   - README는 Gen4 선속도에 가까운 P2P 결과를 보고한다. 단방향 27.5GB/s, 양방향 50.4GB/s, 지연 시간 0.37-0.45마이크로초다.
3. 가이드에는 시행착오로 얻은 설정 세부사항이 포함된다.
   - BIOS 설정은 분기(bifurcation), 강제 Gen4 링크 속도, ASPM, Re-Size BAR, SR-IOV, preferred IO를 다룬다.
   - 커널 메모는 `iommu=off`와 `amd_iommu=off`를 권장한다. 그렇지 않으면 다중 GPU P2P에서 NCCL이 멈출 수 있기 때문이다.
   - ACS는 런타임에 비활성화해 P2P 트래픽이 CPU 루트 포트를 거치지 않고 스위치 패브릭 안에 머물게 한다.
4. 운영 흐름은 로컬이지만 도구 의존도가 높다.
   - 모델 가중치(model weights)는 복제된 ZFS 저장소에 로컬로 보관된다.
   - 러너 디렉터리는 Docker Compose를 사용해 캐시된 로컬 가중치에서 각 모델을 서빙한다.
   - 워크플로는 `opencode` 세션을 내부 추론 머신에 연결하고, 탐색·검색·Telegram 알림·사설 Gitea 협업을 위한 로컬 도구를 사용한다.

### 6. Closing the Verification Loop

1. 이 문서는 에이전트가 엔지니어링 비용을 빌드에서 "아는 것"으로 이동시킨다고 주장한다.
   - 브랜치는 컴파일되고 리뷰를 통과해도 실제 사용자 여정에서는 실패할 수 있다.
   - "검증 루프(verification loop)"는 주장과 증거 사이의 거리로 정의된다.
   - 복합 엔지니어링(compound engineering)은 그 루프를 에이전트가 수행할 수 있는 작업으로 보고, 필요한 곳에서만 사람 판단을 끌어들인다.
2. `/ce-dogfood`는 변경분 범위 브라우저 QA 워크플로로 설명된다.
   - 기준 브랜치 대비 바뀐 부분만 시험하고, 변경분 없는 전체 앱 테스트는 거부한다.
   - 스크린샷이나 추정이 아니라 `agent-browser` CLI를 통해 실제 브라우저를 사용한다.
   - `docs/dogfood-reports/<date>-<branch>-dogfood.md`에 재개 가능한 보고서를 만든다.
3. 워크플로는 페이지가 아니라 흐름에서 시작한다.
   - 행렬을 만들기 전에 진입점, 동작, 분기, 부작용, 실제 종료 상태를 매핑한다.
   - "이메일 규칙(email rule)"은 발송 이벤트만으로는 충분하지 않다고 말한다. 수신자, 링크 대상, 스레드, 스크롤 위치, 내용까지 동작해야 한다.
   - 행렬은 기능 점검, 예외 상태, 오류 상태, 빈 상태, 경험적 점검을 포함한다.
4. 수정 루프는 증거와 에스컬레이션으로 통제된다.
   - 자동 수정은 명확한 버그, 뚜렷한 올바른 수정, 좁은 범위, 큰 제품·아키텍처 트레이드오프가 없는 경우로 제한된다.
   - 실제 수정마다 실패 전과 통과 후가 있는 회귀 테스트가 필요하거나, 브라우저 재현이 더 나은 증거인 이유를 문서화해야 한다.
   - 준비 완료란 녹색 행렬, 녹색 테스트 스위트, 수정 증거, 루프가 닫지 못한 것에 대한 지속 가능한 사람 결정 항목을 뜻한다.

### 7. Getting started with loops

1. ClaudeDevs는 루프를 중단 조건까지 반복되는 에이전트 작업으로 정의한다.
   - 글은 루프를 트리거, 중단 규칙, Claude Code 기본 기능, 작업 적합성으로 구분한다.
   - 모든 작업에 복잡한 루프가 필요한 것은 아니라고 경고한다.
   - 권장 출발점은 작업에 맞는 가장 단순한 루프다.
2. 턴 기반 루프와 목표 기반 루프는 상호작용 작업을 다룬다.
   - 턴 기반 루프는 사용자 프롬프트로 시작되고, Claude가 작업을 끝냈다고 판단하거나 추가 맥락이 필요하다고 판단할 때 멈춘다.
   - 검증 스킬은 수동 점검 절차를 코드화해 Claude가 자기 작업을 더 많이 직접 시험하게 할 수 있다.
   - `/goal`은 점수 기준을 정해진 시도 횟수 안에 달성하는 것처럼 성공 조건과 턴 한도를 사용자가 지정하게 한다.
3. 시간 기반 루프와 선제 루프는 반복 작업이나 외부 시스템을 다룬다.
   - `/loop`는 사용자 컴퓨터에서 프롬프트를 주기적으로 다시 실행한다.
   - `/schedule`은 사용 가능할 때 반복 작업을 클라우드 루틴으로 옮긴다.
   - 선제 루프는 버그 보고나 의존성 업그레이드 같은 흐름을 위해 일정, `/goal`, 동적 워크플로, 자동 모드를 조합할 수 있다.
4. 코드 품질과 사용량 경계도 루프 설계의 일부다.
   - 깨끗한 코드베이스, 접근 가능한 문서, 검증 스킬, 두 번째 에이전트 리뷰가 루프 결과를 개선한다.
   - 적절한 모델 선택, 중단 기준 정의, 결정적 스크립트 사용, 작은 범위 파일럿으로 토큰 사용을 관리한다.
   - `/usage`, `/goal`, `/workflows`는 토큰 사용을 살피고 runaway 작업을 멈추는 방법으로 제시된다.

### 8. A global workspace in language models

1. Anthropic은 Claude 안의 작은 작업공간형 구조를 보고한다.
   - 연구진은 Jacobian 기반 기법인 J-렌즈(J-lens)로 발견했기 때문에 이를 J-공간이라고 부른다.
   - J-공간 패턴은 모델이 말할 수도 있는 단어와 연결되지만, Claude가 조용히 생각하는 개념도 나타낼 수 있다.
   - Anthropic은 이 구조가 명시적으로 프로그래밍된 것이 아니라 훈련 중 나타났다고 말한다.
2. J-공간은 전역 작업공간과 비슷한 기능적 속성을 가진다.
   - Claude는 무엇을 생각하는지 질문받으면 J-공간 내용을 보고할 수 있다.
   - Claude는 과일을 생각하거나 암산을 하는 것처럼 요청에 따라 J-공간 내용을 조절할 수 있다.
   - J-공간 표상은 한 나라를 수도, 언어, 대륙, 통화 질문 전반에서 바꾸는 것처럼 여러 후속 작업에 유연하게 쓰일 수 있다.
3. 이 작업공간은 일상적 유창성보다 고차 인지에 중요한 것으로 보인다.
   - J-공간 내용을 제거해도 유창한 말하기, 감정 분류, 객관식 답변, 지문 사실 검색은 대체로 유지된다.
   - 다단계 추론, 요약, 운율시 작성 성능은 크게 떨어진다.
   - 스페인어를 프랑스어로 바꾸는 예시는 언어를 이름 붙이거나 추론해야 하는 작업은 바꾸지만, 스페인어 자동 이어쓰기는 바꾸지 않는다.
4. Anthropic은 안전 활용과 철학적 한계를 함께 강조한다.
   - J-렌즈 판독값은 평가 상황 인식, 조작 의도, 프롬프트 주입 의심, 조작된 데이터 행동, 숨은 목표를 드러낼 수 있다.
   - 반사실적 성찰 훈련(counterfactual reflection training)은 모델이 중간에 질문받았을 때 무엇을 말할지를 바꾸면 작업 중 내부에서 무엇이 켜지는지도 바뀔 수 있음을 시사한다.
   - 이 연구는 현상적 의식을 증명하지 않지만, 의도적으로 접근 가능한 처리와 자동 처리를 기능적으로 구분할 근거를 제공한다.
