## 📝 Detailed Notes

### 1. You still have to refactor, even with AI

1. Adam Jacob frames the debate as a false binary about whether agents are either magic or useless.
   - He opens by describing two bad internet takes: one that AI-built software is inevitably junk, and another that non-AI engineers are simply behind the curve.
   - His actual claim is narrower and more durable: AI changes where engineering effort sits, but it does not erase that effort.
   - The pressure shifts upward from line-by-line authorship toward architecture, system shape, and the cost of evolving a codebase.
2. The article says the real failure mode is expecting faster generation to remove the need for refactoring.
   - Jacob writes that once software stops fitting the work, the old word still applies: refactoring.
   - Early in his agent use, he found that increasingly complex prototypes collapsed into brittle, incoherent systems.
   - That experience looked like proof that the hype was wrong, but he later reframed it as evidence that his process had not caught up to the tooling.
3. His main mindset shift is from “the agent augments me” to “I augment the agent.”
   - He argues that experienced engineers will often outperform an agent if the workflow is designed only to mimic the engineer directly.
   - The better job, in his view, is to build a machine around the model with constraints, guidance, and reinforcement that reliably produce acceptable output.
   - That means using expertise to shape prompts, skills, checks, and workflows instead of trying to force a non-deterministic model to guess perfectly up front.
4. The S3 example shows why this is still ordinary engineering, just faster.
   - While adding DigitalOcean's S3-compatible object storage support to Swamp, he found the initial agent-generated implementation was brittle in manual testing.
   - The root problem was structural: the codebase created an S3 client in ten different places, and the agent only updated six of them.
   - The fix was not “prompt harder,” but to find the seam where the code was factored wrong and tell the agent to refactor around that seam.
5. The workflow implication is that teams need adversarial loops, architecture skills, and review processes built for agent speed.
   - Jacob says traditional processes broke because they depended on humans learning the code through the friction of editing it directly.
   - He suggests replacing that with mechanisms such as architecture guidance and autonomous review passes that search for simple DRY refactors.
   - His bottom line is conservative: the fundamentals of software engineering still apply, but they now need to happen more quickly and more explicitly around the agent.

### 3. lat.md: A knowledge graph for your codebase, written in markdown

1. `lat.md` starts from a narrow complaint: one flat `AGENTS.md` file stops scaling as projects get larger.
   - The README says a single monolithic document may work for a small project but becomes impractical as decisions and business logic accumulate.
   - Its critique is that important rationale gets buried, undocumented areas grow, and agents begin hallucinating context they should have been able to retrieve.
   - The project therefore treats documentation structure, not just documentation volume, as the real problem to solve.
2. The core proposal is to store project knowledge as a graph of markdown files rooted in `lat.md/`.
   - Sections connect with wiki links like `[[file#Section]]`, which turns documentation into a navigable network instead of a long note.
   - The graph can also link directly into code symbols such as `[[src/auth.ts#validateToken]]`.
   - That makes architecture, domain concepts, and implementation references addressable in a way that agents can traverse consistently.
3. The workflow depends on backlinks from source code into the knowledge graph and on automated consistency checks.
   - The README uses `// @lat: [[section-id]]` style comments to tie source files back to conceptual sections.
   - `lat check` then verifies referential consistency so the graph and code do not silently drift apart.
   - The project extends the same model to tests, where specs can require explicit code mentions and missing backlinks get flagged.
4. The CLI is designed as both a navigation layer and an agent harness.
   - `lat init` scaffolds the directory and sets up agent instructions and hooks for keeping it current.
   - Other commands such as `lat section`, `lat refs`, `lat locate`, and `lat search` help agents and humans retrieve context without defaulting to endless `grep`.
   - `lat expand` can also resolve linked references into prompt-ready context, and `lat mcp` exposes the system through an MCP server for editor integration.
5. The project is opinionated about prerequisites and where its advanced features come from.
   - Semantic search requires either an OpenAI API key or a Vercel AI Gateway key.
   - The README says key resolution can come from env vars, a file path, a helper command, or config written by `lat init`.
   - For local development, the repository expects Node.js 22+ and pnpm, which signals that the tool is aimed at relatively modern JavaScript environments.

   - `[[file#Section]]` 같은 위키 링크를 사용해 문서를 긴 한 장짜리 노트가 아니라 탐색 가능한 네트워크로 만듭니다.
   - 동시에 `[[src/auth.ts#validateToken]]`처럼 코드 심볼로 직접 연결할 수도 있습니다.
   - 이렇게 하면 아키텍처, 도메인 개념, 구현 위치가 모두 주소 가능한 상태가 되어 에이전트가 일관되게 탐색할 수 있습니다.
3. 이 워크플로의 핵심은 소스 코드에서 지식 그래프로 되돌아가는 백링크와 자동 일관성 검사입니다.
   - README는 `// @lat: [[section-id]]` 스타일 주석으로 소스 파일을 개념 섹션과 연결합니다.
   - `lat check`는 그 뒤 참조 일관성을 검사해 그래프와 코드가 조용히 어긋나는 일을 막습니다.
   - 같은 모델은 테스트에도 적용되며, 명세 섹션에 코드 언급을 요구하고 빠진 백링크를 검사할 수 있습니다.
4. CLI는 단순한 유틸리티가 아니라 탐색 레이어이자 에이전트 하네스로 설계돼 있습니다.
   - `lat init`은 디렉터리를 만들고, 이를 최신 상태로 유지할 수 있도록 에이전트용 지침과 훅을 설정합니다.
   - `lat section`, `lat refs`, `lat locate`, `lat search` 같은 명령은 사람과 에이전트가 무한한 `grep`에 빠지지 않고도 필요한 문맥을 찾게 해 줍니다.
   - 또 `lat expand`는 링크된 참조를 프롬프트용 컨텍스트로 펼쳐 주고, `lat mcp`는 에디터 통합을 위한 MCP 서버를 제공합니다.
5. 프로젝트는 필요한 전제조건과 고급 기능의 기반도 분명하게 드러냅니다.
   - 시맨틱 검색을 쓰려면 OpenAI API 키나 Vercel AI Gateway 키가 필요합니다.
   - 키는 환경변수, 파일 경로, 헬퍼 명령, 또는 `lat init`이 저장한 설정에서 순서대로 읽어올 수 있다고 README는 설명합니다.
   - 로컬 개발 환경은 Node.js 22+와 pnpm을 요구하므로, 이 도구가 비교적 최신 자바스크립트 환경을 전제로 한다는 점도 분명합니다.
