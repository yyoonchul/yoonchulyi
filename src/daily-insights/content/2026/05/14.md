### 1. Using Claude Code: The Unreasonable Effectiveness of HTML

1. The article starts from a change in how agents communicate with users.
   - Markdown has become the default agent output because it is portable, editable, and supports basic rich text.
   - Thariq argues that stronger agents now generate larger specs, plans, reports, and artifacts that exceed Markdown’s comfortable reading range.
   - The practical problem is not only generation quality but whether humans actually read and use the generated artifact.
2. HTML is presented as a higher-bandwidth output surface.
   - HTML can represent tables, CSS layouts, SVG diagrams, images, scripts, workflows, and spatial relationships.
   - This lets agents encode richer information without relying on ASCII diagrams or awkward text approximations.
   - A browser-rendered document can be navigated with tabs, responsive layouts, illustrations, links, and visual hierarchy.
3. The argument depends on the changing role of human editing.
   - The author says he increasingly uses agent output as specs, reference files, brainstorming material, and reports rather than manually editing every line.
   - When edits are needed, he often asks Claude to edit the artifact, which removes one of Markdown’s largest advantages.
   - HTML’s noisier diffs remain a downside, but the author views readability and engagement as more important in many workflows.
4. The use cases extend beyond static documents.
   - For planning, agents can create exploration grids, mockups, code snippets, data-flow diagrams, and implementation plans.
   - For code review, HTML can render diffs, annotations, flowcharts, module maps, and severity-coded findings.
   - For design and prototypes, HTML can provide sliders, knobs, copy buttons, animation previews, and parameter export.
5. The article ends by treating HTML as a way to keep humans in the loop.
   - Custom one-off editors can make hard-to-express decisions visible, such as ticket triage, config changes, prompt tuning, dataset labeling, or visual value selection.
   - The suggested entry point is simple prompting, such as asking Claude to make an HTML file or artifact for the specific situation.
   - The core claim is that richer artifacts make agent work easier to inspect, share, and guide.

### 4. If AI Writes Your Code, Why Use Python?

1. The essay argues that the old language-choice bargain is breaking.
   - Python and TypeScript won many projects because they were fast to ship, had large ecosystems, and were easy to hire for.
   - Rust, Go, C++, and similar languages offered performance but carried higher human learning and tooling costs.
   - Mitchem’s thesis is that AI agents reduce the human-cost side of that tradeoff.
2. Strong typing and compiler feedback become agent advantages.
   - The essay argues that agents benefit from tight compile-check loops, explicit types, and precise error messages.
   - Rust, Go, and Swift are presented as languages where agents can self-correct more effectively than in looser environments.
   - The systems languages that were hard for humans can be easier for agents because the feedback loop is clearer.
3. The article points to recent systems-language wins.
   - Microsoft’s TypeScript compiler rewrite in Go is cited as a major example of choosing a faster language for flagship tooling.
   - Nicholas Carlini’s Claude-agent C compiler in Rust is cited as a case where agent orchestration changes the scale of what one researcher can attempt.
   - Other examples include Steve Klabnik’s Rue work and Ladybird JavaScript engine porting to Rust.
4. Ecosystem arguments are also weakening.
   - The essay notes that many Python performance layers already use Rust, including pydantic-core, Polars, tokenizers, and orjson.
   - It cites Rust-based tools such as ruff, uv, and ty as examples of infrastructure under Python becoming systems-language infrastructure.
   - The point is that Python and JavaScript increasingly wrap faster languages rather than being the entire implementation surface.
5. The future is not presented as a clean sweep.
   - Mitchem notes that old answers can still be right, including Prisma’s move away from a Rust query engine in favor of TypeScript/WASM.
   - PyTorch remains dominant in deep-learning research, and AI is not equally good at every systems language.
   - The durable claim is narrower: language choice will increasingly favor what agents can reliably build, not only what humans find easiest to type.

### 5. SkillOS: Learning Skill Curation for Self-Evolving Agents

1. The paper targets a common agent limitation.
   - LLM agents increasingly handle streaming tasks, but many still behave as one-off solvers.
   - Reusable skills distilled from experience are presented as the substrate for self-evolving agents.
   - The bottleneck is not only skill execution but high-quality skill curation over time.
2. SkillOS separates execution from curation.
   - A frozen agent executor retrieves and applies existing skills.
   - A trainable skill curator updates an external SkillRepo based on accumulated experience.
   - This division lets the system study how to improve the skill repository without constantly retraining the executor.
3. The training signal is designed for delayed effects.
   - The paper argues that existing approaches rely on manual curation, heuristic operations, or short-horizon training.
   - SkillOS uses composite rewards and grouped task streams based on skill-relevant dependencies.
   - Earlier trajectories update the SkillRepo, while later related tasks evaluate whether those updates helped.
4. The reported results cover multiple task styles.
   - The authors report consistent gains over memory-free baselines and strong memory-based baselines.
   - The experiments include multi-turn agentic tasks and single-turn reasoning tasks.
   - The learned curator reportedly generalizes across executor backbones and task domains.
5. The qualitative finding is that skills become richer artifacts.
   - The paper says learned curation produces more targeted skill use.
   - SkillRepo entries evolve into more structured Markdown files over time.
   - Those files encode higher-level meta-skills rather than only narrow task fragments.

### 9. Interaction Models: A Scalable Approach to Human-AI Collaboration

1. Thinking Machines Lab argues that current interfaces push humans out of the loop.
   - The post says many AI labs optimize for autonomous long-running work.
   - In real work, users often cannot specify everything upfront and need to clarify, interrupt, and guide.
   - The limitation is framed as an interface bandwidth problem, not only a model intelligence problem.
2. Interaction models move beyond turn-based interfaces.
   - Current systems often wait for the user to finish speaking or typing, then freeze perception while generating.
   - Thinking Machines wants AI to continuously perceive and respond across audio, video, and text.
   - The goal is collaboration that resembles talking, showing, listening, interjecting, and acting in real time.
3. Interactivity is trained into the model rather than bolted on.
   - The post criticizes harness-based systems that stitch together components for voice activity, interruption, multimodality, or concurrency.
   - It argues that interactivity should scale with intelligence and therefore should be part of the model itself.
   - Capabilities include implicit turn tracking, jumping in contextually, concurrent speech, elapsed-time awareness, search, tool use, and generative UI.
4. The architecture combines fast interaction with background reasoning.
   - The interaction model uses micro-turns, processing and generating roughly 200ms slices.
   - A background model runs asynchronous deeper reasoning, tool use, search, or long-horizon work.
   - Both systems share context so the interaction model can remain present while longer work proceeds.
5. The benchmarks focus on time and visual interaction.
   - TML-interaction-small scores 64.7 on TimeSpeak versus 4.3 for GPT realtime-2.0 minimal.
   - It scores 81.7 on CueSpeak versus 2.9, and 35.4 on RepCount-A off-by-one versus 1.3.

### 10. A History of IDEs at Google

1. The article starts with Google’s historically fragmented IDE culture.
   - Le Brun says Google’s main monorepo had strict tooling and conventions, but IDE choice remained an exception.
   - In 2011, senior engineers were asked whether a uniform IDE for Googlers was possible.
   - The prevailing answer was no, because editor preferences are deeply personal.
2. Fragmentation created duplicated integration work.
   - Engineers could choose their own IDEs, but Google-specific integrations had to be rebuilt repeatedly.
   - Examples include Bazel support, Starlark tooling, formatters, and code search.
   - Google’s internal culture made this manageable, but it was still inefficient from a company-productivity perspective.
3. Cider solved Google-specific backend problems.
   - Cider was a light client that opened quickly in the browser.
   - Its backend indexed the whole codebase so code intelligence was available on demand.
   - The backend had to connect identifiers, types, and references across a massive language graph updated at high commit velocity.
4. Cider V adopted VS Code as the frontend.
   - In 2020, the team chose VS Code because it was mature, language-agnostic, extensible, and web-based.
   - This let Google inherit a mature editor and extension ecosystem while keeping its specialized backend.
   - The migration still took years because version control, code review, completion, refactoring, extension delivery, and workflow polish all needed integration.
5. The result was not fully uniform but became dominant.
   - By 2023, 80% of development in Google’s main codebase happened in Cider V.
   - Cider V won through deep integration with company tools, including version control and inline code-review comments.
   - The article’s larger lesson is that editor standardization can emerge from superior integrations even when universal preference agreement is impossible.

### 11. Build iterative repair loops with Codex

1. The Cookbook example teaches a closed-loop agent workflow.
   - The loop produces an output, validates it, and uses feedback to improve the next pass.
   - The worked example repairs stale or broken OpenAI API and SDK notebooks.
   - Codex reviews the artifact, applies focused changes, validates, and repeats when evidence shows remaining issues.
2. The workflow separates review, repair, and validation.
   - Review inspects the artifact and returns structured findings without editing files.
   - Repair receives findings plus validation feedback and edits only the copied artifact.
   - Validation executes the notebook and judges whether the repaired version satisfies the defined cases.
3. Shared business rules keep the loop grounded.
   - The example explicitly modernizes stale APIs such as chat completions to Responses and legacy function-calling schemas to current tools.
   - It also updates qdrant search calls and old Evals CLI examples.
   - Reader-experience rules require fresh-environment setup, local runnable samples, explicit prerequisites, and preservation of the teaching goal.
4. Structured outputs make the loop auditable.
   - Findings include artifact, issue type, severity, description, and suggested fix direction.
   - Repair results include change summaries, unresolved items, and updated artifact paths.
   - Validation results include execution evidence, scoring, remaining deltas, and pass/fail status.
5. The pattern generalizes beyond notebooks.
   - The Cookbook suggests applying the same loop to regulated content, support articles, protocols, deprecated APIs, and codebases.
   - A good production loop stops when validation passes, attempts are exhausted, deltas stop changing, or human review is needed.
   - The main idea is that agentic maintenance becomes more trustworthy when every pass is tied to evidence.

### 12. karpathy/autoresearch

1. The repository explores autonomous AI research on a small real training setup.
   - The project gives an AI agent a simplified single-GPU nanochat training environment.
   - The agent runs experiments overnight instead of requiring the human to directly edit research code.
   - The hoped-for output is a log of experiments and a better model by morning.
2. The workflow is intentionally constrained.
   - The repo has a few key files: `prepare.py`, `train.py`, and `program.md`.
   - `prepare.py` handles fixed constants, data preparation, tokenizer training, dataloading, and evaluation.
   - `train.py` is the file the agent modifies, including the GPT model, optimizer, hyperparameters, architecture, and training loop.
3. The metric and budget make experiments comparable.
   - Each training run uses a fixed 5-minute wall-clock budget, excluding startup and compilation.
   - The metric is validation bits per byte, where lower is better.
   - This metric is intended to be vocab-size-independent so architectural changes can be compared more fairly.
4. Humans program the research organization rather than every code change.
   - The README says the core idea is that humans do not touch Python files in the usual researcher manner.
   - Instead, humans edit `program.md`, which supplies context and instructions to the agent.
   - The repository treats `program.md` as a baseline that can be iterated to create a better autonomous research organization.
5. The project is a small demonstration of a larger agentic research pattern.
   - It makes agent behavior inspectable by keeping the setup deliberately compact.
   - It separates stable environment code from agent-editable experiment code.
   - The broader implication is that research progress may depend on designing agent instructions, evaluation loops, and experiment-selection policies as much as writing model code directly.

### 1. Using Claude Code: The Unreasonable Effectiveness of HTML

1. 글은 에이전트가 사용자와 소통하는 방식의 변화에서 출발한다.
   - Markdown은 이식성이 좋고 편집 가능하며 기본적인 리치 텍스트를 지원하기 때문에 기본 에이전트 출력이 됐다.
   - Thariq는 더 강한 에이전트가 Markdown으로 읽기 불편한 큰 스펙, 계획, 보고서, 산출물을 만들고 있다고 본다.
   - 실질적 문제는 생성 품질만이 아니라 사람이 그 산출물을 실제로 읽고 활용하느냐다.
2. HTML은 더 높은 대역폭의 출력 표면으로 제시된다.
   - HTML은 표, CSS 레이아웃, SVG 다이어그램, 이미지, 스크립트, 워크플로, 공간 관계를 표현할 수 있다.
   - 에이전트는 ASCII 다이어그램이나 어색한 텍스트 근사에 기대지 않고 더 풍부한 정보를 담을 수 있다.
   - 브라우저 렌더링 문서는 탭, 반응형 레이아웃, 그림, 링크, 시각적 위계로 탐색할 수 있다.
3. 이 주장은 인간 편집의 역할 변화에 의존한다.
   - 작성자는 에이전트 출력을 직접 편집할 문장보다 스펙, 참고 파일, 브레인스토밍 자료, 보고서로 더 자주 쓴다고 말한다.
   - 수정이 필요할 때도 Claude에게 산출물을 고치라고 요청하는 경우가 많아 Markdown의 큰 장점 하나가 약해진다.
   - HTML diff가 지저분하다는 단점은 남지만, 많은 워크플로에서는 가독성과 참여도가 더 중요하다고 본다.
4. 사용 사례는 정적 문서에 머물지 않는다.
   - 계획 단계에서는 에이전트가 탐색 그리드, 목업, 코드 조각, 데이터 흐름도, 구현 계획을 만들 수 있다.
   - 코드 리뷰에서는 diff, 주석, 플로차트, 모듈 지도, 심각도별 지적 사항을 렌더링할 수 있다.

### 4. If AI Writes Your Code, Why Use Python?

1. 글은 오래된 언어 선택의 거래가 깨지고 있다고 주장한다.
   - Python과 TypeScript는 빠른 출시, 큰 생태계, 채용 용이성 때문에 많은 프로젝트에서 이겼다.
   - Rust, Go, C++ 같은 언어는 성능을 제공했지만 학습과 도구 비용이 컸다.
   - Mitchem의 논지는 AI 에이전트가 이 거래에서 인간 비용 부분을 낮춘다는 것이다.
2. 강한 타입과 컴파일러 피드백은 에이전트의 장점이 된다.
   - 글은 에이전트가 촘촘한 컴파일-검사 루프, 명시적 타입, 정확한 오류 메시지에서 이익을 얻는다고 주장한다.
   - Rust, Go, Swift는 에이전트가 느슨한 환경보다 더 효과적으로 자기 수정할 수 있는 언어로 제시된다.
   - 인간에게 어려웠던 시스템 언어가 피드백 루프가 명확하기 때문에 에이전트에게는 더 쉬울 수 있다.
3. 글은 최근 시스템 언어 승리를 사례로 든다.
   - Microsoft의 TypeScript 컴파일러 Go 재작성은 핵심 도구에 더 빠른 언어를 선택한 주요 사례로 언급된다.
   - Nicholas Carlini의 Claude 에이전트 기반 Rust C 컴파일러는 에이전트 오케스트레이션이 개인 연구자의 시도 규모를 바꾸는 사례로 제시된다.
   - Steve Klabnik의 Rue 작업과 Ladybird JavaScript 엔진의 Rust 포팅도 함께 언급된다.
4. 생태계 논리도 약해지고 있다.
   - 글은 pydantic-core, Polars, tokenizers, orjson처럼 많은 Python 성능 계층이 이미 Rust를 쓴다고 지적한다.
   - ruff, uv, ty 같은 Rust 기반 도구는 Python 아래의 인프라가 시스템 언어 인프라가 되고 있음을 보여준다.
   - 요점은 Python과 JavaScript가 전체 구현 표면이라기보다 더 빠른 언어를 감싸는 계층이 되고 있다는 것이다.
5. 미래는 완전한 승리로 제시되지 않는다.
   - Mitchem은 Prisma가 Rust 쿼리 엔진을 TypeScript/WASM 중심으로 바꾼 것처럼 기존 답이 여전히 맞을 수 있다고 인정한다.
   - PyTorch는 딥러닝 연구에서 여전히 지배적이고, AI가 모든 시스템 언어에 똑같이 강한 것도 아니다.
   - 지속되는 주장은 더 좁다. 언어 선택은 사람이 타이핑하기 쉬운 것뿐 아니라 에이전트가 안정적으로 만들 수 있는 것을 더 많이 따르게 된다.

### 5. SkillOS: Learning Skill Curation for Self-Evolving Agents

1. 논문은 흔한 에이전트 한계를 겨냥한다.
   - LLM 에이전트는 스트리밍 작업을 점점 더 많이 처리하지만, 여전히 많은 경우 일회성 문제 해결자로 동작한다.
   - 경험에서 추출한 재사용 가능한 스킬은 자기 진화 에이전트의 기반으로 제시된다.
   - 병목은 스킬 실행뿐 아니라 시간이 지날수록 좋은 스킬을 큐레이션하는 능력이다.
2. SkillOS는 실행과 큐레이션을 분리한다.
   - 고정된 agent executor는 기존 스킬을 검색하고 적용한다.
   - 학습 가능한 skill curator는 누적 경험을 바탕으로 외부 SkillRepo를 업데이트한다.
   - 이 분리는 executor를 계속 재학습하지 않고 스킬 저장소 개선 방법을 연구하게 해준다.
3. 학습 신호는 지연 효과를 다루도록 설계된다.
   - 논문은 기존 접근이 수동 큐레이션, 휴리스틱 연산, 짧은 지평의 학습에 의존한다고 비판한다.
   - SkillOS는 복합 보상과 스킬 관련 의존성에 따라 묶인 작업 스트림을 사용한다.
   - 앞선 trajectory가 SkillRepo를 업데이트하고, 이후 관련 작업이 그 업데이트가 도움이 됐는지 평가한다.
4. 보고된 결과는 여러 작업 형태를 포괄한다.
   - 저자들은 메모리 없는 기준선과 강한 메모리 기반 기준선 모두보다 일관되게 낫다고 보고한다.
   - 실험은 멀티턴 에이전트 작업과 싱글턴 추론 작업을 포함한다.
   - 학습된 curator는 다른 executor backbone과 작업 도메인에도 일반화된다고 한다.
5. 정성적 발견은 스킬이 더 풍부한 artifact가 된다는 점이다.
   - 논문은 학습된 큐레이션이 더 표적화된 스킬 사용을 만든다고 말한다.
   - SkillRepo 항목은 시간이 지나며 더 구조화된 Markdown 파일로 진화한다.
   - 그 파일들은 좁은 작업 조각만이 아니라 더 높은 수준의 메타 스킬을 인코딩한다.

### 6. Ask LLMs for HTML output

1. Karpathy의 실용적 권고는 단순하다.
   - 그는 LLM 질의 끝에 응답을 HTML로 구성해 달라고 요청하라고 제안한다.
   - 생성된 파일은 브라우저에서 열어 더 시각적인 결과로 볼 수 있다.
   - 그는 모델에게 슬라이드쇼와 유사한 발표 형식을 만들라고 요청해도 성공한 적이 있다고 말한다.
2. 글은 입력과 출력 채널을 다르게 본다.
   - Karpathy는 오디오가 사람이 AI에 입력하기 선호하는 채널이라고 말한다.
   - 그는 이미지, 애니메이션, 비디오를 포함한 시각이 AI 출력의 선호 채널이라고 주장한다.
   - 이유는 인간 인지의 큰 부분이 시각 처리와 병렬적인 시각 입력에 맞춰져 있기 때문이다.
3. HTML은 초기 인터페이스 단계로 배치된다.
   - 진행은 원문 텍스트, Markdown, HTML 순서로 설명된다.
   - Markdown은 제목, 굵게, 기울임, 표를 더하지만 여전히 텍스트에 가깝다.
   - HTML은 절차적 구조를 유지하면서도 더 풍부한 그래픽, 레이아웃, 상호작용을 가능하게 한다.
4. 장기 인터페이스 비전은 더 야심적이다.
   - Karpathy는 출력이 대화형 neural video나 simulation으로 이동할 수 있다고 추측한다.
   - 그는 정확한 절차적 소프트웨어 artifact와 diffusion류 시스템이 만드는 neural artifact를 구분한다.
   - 이 두 artifact를 어떻게 결합할지에 대해서는 많은 열린 질문이 남아 있다.
5. 입력도 여전히 크게 개선되어야 한다.
   - 그는 오디오, 텍스트, 비디오만으로는 충분하지 않다고 말한다.
   - 컴퓨터 옆에 사람이 함께 있을 때처럼 화면의 대상에 포인팅하고 제스처하는 기능이 필요하다고 본다.
   - 더 넓은 논지는 brain-computer interface로 곧장 가기 전에 인간-AI 입출력 결합에는 큰 개선 여지가 있다는 것이다.

### 9. Interaction Models: A Scalable Approach to Human-AI Collaboration

1. Thinking Machines Lab은 현재 인터페이스가 사람을 루프 밖으로 밀어낸다고 주장한다.
   - 글은 많은 AI 연구소가 자율적인 장시간 작업을 최적화한다고 말한다.
   - 실제 작업에서는 사용자가 모든 요구사항을 미리 지정할 수 없고 명확화, 중단, 지도가 필요하다.
   - 한계는 모델 지능만이 아니라 인터페이스 대역폭 문제로 설명된다.
2. Interaction model은 턴 기반 인터페이스를 넘어선다.
   - 현재 시스템은 사용자가 말하거나 입력을 끝낼 때까지 기다리고, 생성 중에는 지각이 멈추는 경우가 많다.
   - Thinking Machines는 AI가 오디오, 비디오, 텍스트 전반에서 계속 지각하고 반응하기를 원한다.
   - 목표는 말하기, 보여주기, 듣기, 끼어들기, 행동하기가 실시간으로 이어지는 협업이다.
3. 상호작용성은 외부 장치가 아니라 모델에 학습된다.
   - 글은 음성 활동, 중단, 멀티모달, 동시성을 여러 컴포넌트로 엮는 harness 기반 시스템을 비판한다.
   - 상호작용성은 지능과 함께 스케일해야 하므로 모델 자체의 일부여야 한다고 주장한다.
   - 능력에는 암묵적 턴 추적, 맥락적 끼어들기, 동시 발화, 경과 시간 인식, 검색, 도구 사용, generative UI가 포함된다.
4. 아키텍처는 빠른 상호작용과 백그라운드 추론을 결합한다.
   - interaction model은 약 200ms 단위의 입력과 출력을 처리하고 생성하는 micro-turn을 사용한다.
   - background model은 더 깊은 추론, 도구 사용, 검색, 장기 작업을 비동기로 실행한다.
   - 두 시스템은 context를 공유하므로 긴 작업이 진행되는 동안 interaction model이 계속 사용자와 함께 있을 수 있다.
5. 벤치마크는 시간과 시각 상호작용에 초점을 둔다.
   - TML-interaction-small은 TimeSpeak에서 64.7점을 받아 GPT realtime-2.0 minimal의 4.3점을 크게 앞선다.
   - CueSpeak에서는 81.7 대 2.9, RepCount-A off-by-one에서는 35.4 대 1.3을 기록한다.
   - ProactiveVideoQA와 Charades에서도 기준선을 앞서지만, 글은 긴 세션, 저지연 배포, 안전성의 한계를 함께 언급한다.

### 10. A History of IDEs at Google

1. 글은 Google의 역사적으로 분절된 IDE 문화에서 시작한다.
   - Le Brun은 Google의 메인 모노레포에는 엄격한 도구와 규칙이 있었지만 IDE 선택은 예외였다고 말한다.
   - 2011년 시니어 엔지니어들은 Googler를 위한 통일 IDE가 가능한지 질문을 받았다.
   - 당시 지배적 답은 에디터 선호가 매우 개인적이기 때문에 불가능하다는 것이었다.
2. 분절은 통합 작업의 중복을 만들었다.
   - 엔지니어들은 원하는 IDE를 선택할 수 있었지만 Google 특화 통합은 반복 구현되어야 했다.
   - 예시에는 Bazel 지원, Starlark 도구, 포맷터, 코드 검색이 포함된다.
   - Google의 내부 문화가 이를 감당하게 했지만, 회사 생산성 관점에서는 여전히 비효율적이었다.
3. Cider는 Google 특화 백엔드 문제를 풀었다.
   - Cider는 브라우저에서 빠르게 열리는 가벼운 클라이언트였다.
   - 백엔드는 전체 코드베이스를 인덱싱해 필요한 순간 코드 인텔리전스를 제공했다.
   - 백엔드는 매우 빠른 commit 속도로 갱신되는 거대한 언어 그래프에서 식별자, 타입, 참조를 연결해야 했다.
4. Cider V는 VS Code를 프런트엔드로 채택했다.
   - 2020년 팀은 VS Code가 성숙하고, 언어 중립적이며, 확장 가능하고, 웹 기반이기 때문에 선택했다.
   - Google은 특화 백엔드를 유지하면서 성숙한 에디터와 확장 생태계를 가져올 수 있었다.
   - 그래도 버전 관리, 코드 리뷰, 자동완성, 리팩터링, 확장 배포, 워크플로 polish가 모두 통합되어야 했기 때문에 전환에는 수년이 걸렸다.
5. 결과는 완전한 통일은 아니었지만 지배적 채택이었다.
   - 2023년에는 Google 메인 코드베이스 개발의 80%가 Cider V에서 일어났다.
   - Cider V는 버전 관리와 inline 코드 리뷰 댓글을 포함한 회사 도구와의 깊은 통합으로 선택받았다.
   - 글의 더 큰 교훈은 모두가 같은 에디터를 좋아하지 않아도 더 나은 통합이 사실상의 표준화를 만들 수 있다는 것이다.

### 11. Build iterative repair loops with Codex

1. Cookbook 예시는 폐쇄 루프 에이전트 워크플로를 가르친다.
   - 루프는 산출물을 만들고, 검증하고, 피드백을 사용해 다음 패스를 개선한다.
   - 예시는 오래되거나 깨진 OpenAI API·SDK 노트북을 수리한다.
   - Codex는 artifact를 검토하고, 초점 있는 변경을 적용하고, 검증한 뒤 남은 문제가 있으면 반복한다.
2. 워크플로는 review, repair, validation을 분리한다.
   - Review는 artifact를 읽고 파일을 수정하지 않은 채 구조화된 발견 사항을 반환한다.
   - Repair는 발견 사항과 검증 피드백을 받아 복사된 artifact만 수정한다.
   - Validation은 노트북을 실행하고 수리된 버전이 정의된 기준을 만족하는지 판단한다.
3. 공유 business rule은 루프를 고정한다.
   - 예시는 chat completions를 Responses로, legacy function-calling schema를 현재 tools schema로 바꾸는 등 오래된 API를 명시적으로 현대화한다.
   - qdrant search 호출과 오래된 Evals CLI 예시도 업데이트한다.
   - 독자 경험 규칙은 fresh environment setup, 로컬 실행 가능 샘플, 명시적 전제 조건, 원래 교육 목표 보존을 요구한다.
4. 구조화된 출력은 루프를 감사 가능하게 만든다.
   - Findings는 artifact, issue type, severity, description, suggested fix direction을 포함한다.
   - Repair 결과는 변경 요약, 미해결 항목, 업데이트된 artifact 경로를 포함한다.
   - Validation 결과는 실행 증거, 점수, 남은 delta, 통과 여부를 포함한다.
5. 이 패턴은 노트북을 넘어 일반화된다.
   - Cookbook은 같은 루프를 규제 콘텐츠, 지원 문서, 프로토콜, deprecated API, 코드베이스에 적용할 수 있다고 제안한다.
   - 좋은 production loop는 검증 통과, 시도 횟수 소진, delta 정체, 인간 리뷰 필요 중 하나에서 멈춘다.
   - 핵심 아이디어는 모든 패스가 증거에 연결될 때 agentic maintenance가 더 신뢰 가능해진다는 것이다.

### 12. karpathy/autoresearch

1. 저장소는 작은 실제 학습 환경에서 자율 AI 연구를 실험한다.
   - 프로젝트는 AI 에이전트에게 단순화된 단일 GPU nanochat 학습 환경을 제공한다.
   - 에이전트는 인간이 연구 코드를 직접 고치는 대신 밤새 실험을 실행한다.
   - 기대 산출물은 아침에 확인할 수 있는 실험 로그와 가능하면 더 나은 모델이다.
2. 워크플로는 의도적으로 제한된다.
   - 저장소의 핵심 파일은 `prepare.py`, `train.py`, `program.md` 몇 개다.
   - `prepare.py`는 고정 상수, 데이터 준비, 토크나이저 학습, dataloader, 평가를 담당한다.
   - `train.py`는 에이전트가 수정하는 파일이며 GPT 모델, optimizer, hyperparameter, architecture, training loop를 포함한다.
3. metric과 예산은 실험 비교를 가능하게 한다.
   - 각 학습 실행은 startup과 compilation을 제외한 5분 고정 wall-clock 예산을 사용한다.
   - metric은 validation bits per byte이며 낮을수록 좋다.
   - 이 metric은 vocabulary size에 독립적이어서 architecture 변경을 더 공정하게 비교하려는 의도를 갖는다.
4. 사람은 모든 코드 변경보다 연구 조직을 프로그래밍한다.
   - README는 핵심 아이디어가 사람이 일반 연구자처럼 Python 파일을 직접 만지지 않는 것이라고 말한다.
   - 대신 사람은 에이전트에게 context와 instruction을 제공하는 `program.md`를 수정한다.
   - 저장소는 `program.md`를 더 나은 자율 연구 조직을 만들기 위해 반복 개선할 수 있는 baseline으로 취급한다.
5. 프로젝트는 더 큰 에이전트 연구 패턴의 작은 시연이다.
   - 설정을 의도적으로 작게 유지해 에이전트 행동을 관찰 가능하게 만든다.
   - 안정적인 환경 코드와 에이전트가 수정하는 실험 코드를 분리한다.
   - 더 넓은 함의는 연구 진전이 모델 코드를 직접 작성하는 것만큼 에이전트 지시, 평가 루프, 실험 선택 정책 설계에 달릴 수 있다는 것이다.
