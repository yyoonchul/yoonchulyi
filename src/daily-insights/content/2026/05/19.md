## 📝 Detailed Notes

### 1. Linus Torvalds says AI-powered bug hunters have made Linux security mailing list ‘almost entirely unmanageable’

1. Torvalds links the mailing-list problem to duplicate AI bug reports.
   - The article says Torvalds raised the issue in his weekly Linux 7.1 release-candidate update.
   - He described kernel development progress as fairly normal, then pointed readers to project documentation.
   - The warning is that many researchers are using similar AI tools to find the same bugs and submit overlapping private reports.
2. The operational burden falls on maintainers rather than on reporters.
   - Maintainers spend time forwarding reports to the right people instead of working on fixes.
   - They also have to explain that a reported issue was already fixed a week or a month earlier.
   - Torvalds frames that loop as pointless churn rather than useful security work.
3. The private security list is a poor fit for AI-discovered issues.
   - Torvalds argues AI-detected bugs are usually not secret by nature.
   - Private reporting makes duplication worse because reporters cannot see one another’s findings.
   - The article’s implication is that public discussion and existing documentation are better coordination points for these cases.
4. Torvalds does not reject AI tools, but rejects shallow use.
   - He says AI tools are useful when they help rather than create make-believe work.
   - The productive path is to read the documentation before filing another report.
   - He asks researchers to add patches or understanding on top of what the AI found.
5. The remarks sit alongside a broader debate inside open source.
   - The Register contrasts Torvalds’s blunt warning with Greg Kroah-Hartman’s view that AI has become more useful to the FOSS community.
   - The difference is less about whether AI can find bugs and more about whether reports arrive with enough context to help maintainers.
   - The article treats AI bug hunting as a workflow design problem, not only as a detection-quality problem.

### 2. Google has sold so much TPU capacity that its own researchers are queueing for the rest

1. Google’s TPU success has created an internal allocation problem.
   - TNW says Alphabet has a strong AI infrastructure stack: cloud business, custom chips, and external TPU supply deals.
   - Bloomberg reporting cited by TNW says Google and Google DeepMind researchers are now competing for compute access.
   - The article frames the problem as a consequence of commercial success rather than a simple infrastructure failure.
2. Anthropic is the largest named source of external TPU demand.
   - Google has agreed to invest up to $40bn in Anthropic, with a deal that includes five gigawatts of TPU capacity over five years.
   - The same arrangement includes access to up to one million seventh-generation Ironwood chips.
   - TNW also cites a Broadcom-mediated line for another 3.5GW of Anthropic TPU capacity from 2027, on top of 1GW in 2026.
3. The shortage is both physical and organizational.
   - DeepMind CEO Demis Hassabis has pointed to supplier constraints around key components.
   - High-bandwidth memory from Samsung, Micron, and SK Hynix is described as a common choke point.
   - The internal allocation problem is different because Google controls how scarce compute is rationed across teams.
4. Google is trying to serve three strategic goals with one fabric.
   - TPUs help Google sell cloud infrastructure to major external customers and validate its alternative to Nvidia.
   - The same infrastructure must support Gemini training and DeepMind research.
   - Meta is named as another commercial-scale TPU customer, which further tightens the available pool.
5. The next question is whether Google can expand capacity fast enough.
   - TNW cites Alphabet’s guided 2026 capex range of $175bn-$185bn and Big Tech AI infrastructure spending above $650bn.
   - Google has said it is bringing more than a gigawatt of new AI compute online in 2026.
   - The article says Google’s version is notable because the company has become a key infrastructure supplier to its own competitors.

### 3. How Claude Code works in large codebases: Best practices and where to start

1. The article says large-codebase performance depends on navigation.
   - Claude Code works best when it has enough starting context to know where to look.
   - Vague searches across extremely large repositories can hit context-window limits before useful work begins.
   - Teams that invest in codebase setup see stronger outcomes because Claude spends less effort discovering structure from scratch.
2. Anthropic defines the harness as the real operating layer around the model.
   - The harness includes `CLAUDE.md` files, hooks, skills, plugins, and MCP servers.
   - LSP integrations and subagents extend the setup after the basic layers are in place.
   - The article argues teams should build these layers in order because each layer depends on context and conventions from the previous one.
3. The recommended context strategy is lean and layered.
   - Root `CLAUDE.md` files should cover the big picture, critical gotchas, and pointers.
   - Subdirectory files should describe local conventions and commands for that part of the repository.
   - Starting Claude in the relevant subdirectory keeps the task scoped while still allowing root context to load through the directory tree.
4. Deterministic tooling should carry rules that prompts should not carry.
   - Hooks can enforce linting, formatting, and other checks more reliably than repeated natural-language reminders.
   - Skills keep specialized workflows available on demand without bloating every session.
   - Plugins package skills, hooks, and MCP configuration so working setups can be distributed rather than remaining tribal knowledge.
5. Adoption requires ownership, not only configuration.
   - Anthropic says successful rollouts often start with a small infrastructure team or a clear DRI for Claude Code configuration.
   - That owner manages settings, permission policy, plugin marketplace choices, and `CLAUDE.md` conventions.
   - In regulated organizations, the article recommends approved skills, required code-review processes, limited initial access, and cross-functional governance.

### 3. How Claude Code works in large codebases: Best practices and where to start

1. 글은 대형 코드베이스 성능이 탐색 가능성에 좌우된다고 말한다.

   - 매우 큰 저장소에서 모호한 패턴을 찾으라고 하면 유용한 작업이 시작되기 전에 컨텍스트 창 한계에 걸릴 수 있다.
   - 코드베이스 설정에 투자한 팀은 Claude가 구조를 처음부터 발견하는 데 드는 노력을 줄이기 때문에 더 나은 결과를 얻는다.
2. Anthropic은 하네스를 모델 주변의 실제 운영 계층으로 정의한다.
   - 하네스는 `CLAUDE.md` 파일, 훅, 스킬, 플러그인, MCP 서버를 포함한다.
   - LSP 통합과 서브에이전트는 기본 계층이 갖춰진 뒤 설정을 확장한다.
   - 글은 각 계층이 앞선 계층의 컨텍스트와 관례에 의존하므로 순서대로 구축해야 한다고 주장한다.
3. 권장 컨텍스트 전략은 간결하고 계층화된 구조다.
   - 루트 `CLAUDE.md`는 전체 그림, 중요한 주의점, 포인터를 다뤄야 한다.
   - 하위 디렉터리 파일은 저장소의 해당 부분에 적용되는 로컬 관례와 명령을 설명해야 한다.
   - 관련 하위 디렉터리에서 Claude를 시작하면 작업 범위를 좁히면서도 디렉터리 트리를 통해 루트 컨텍스트를 함께 불러올 수 있다.
4. 결정론적 도구가 프롬프트가 떠안지 말아야 할 규칙을 맡아야 한다.

   - 플러그인은 스킬, 훅, MCP 설정을 묶어 잘 작동하는 구성을 부족 지식으로 남기지 않고 배포하게 해준다.
5. 도입에는 설정뿐 아니라 소유권이 필요하다.
   - Anthropic은 성공적인 도입이 작은 인프라 팀이나 Claude Code 설정의 명확한 DRI에서 시작되는 경우가 많다고 말한다.
   - 그 소유자는 설정, 권한 정책, 플러그인 마켓플레이스 선택, `CLAUDE.md` 관례를 관리한다.
   - 규제 산업 조직에는 승인된 스킬, 필수 코드 리뷰 절차, 제한된 초기 접근, 교차 기능 거버넌스를 권한다.
